# 《项目管理知识体系指南》（PMBOK）- PMI

## 📖 基本信息
- **来源**: Project Management Institute（PMI）第七版（2021）
- **核心主题**: 项目管理的标准框架——12 项原则 + 8 个绩效域 + 敏捷与预测混合方法
- **目标读者**: 项目经理；需要管理交付风险的 FDE；备考 PMP 认证
- **阅读难度**: intermediate

## 🎯 核心价值
> PMBOK 不是流程手册，是一套思维工具——帮助你在复杂的利益相关者环境中，有序地推进不确定性项目走向成功交付。

## 📚 知识框架

### PMBOK 第七版：从流程到原则

第七版重大转变：从 5 大过程组 + 10 大知识领域，转为 **12 项原则 + 8 个绩效域**

---

### 1. 12 项项目管理原则

| 原则 | 核心要义 |
|------|---------|
| 1. 做负责任的管理者 | 对组织、社会、环境负责 |
| 2. 建立协作团队 | 团队比流程更重要 |
| 3. 与利益相关者互动 | 识别、理解、持续沟通 |
| 4. 聚焦价值 | 交付物 ≠ 价值；关注结果 |
| 5. 识别系统交互 | 项目是系统，组件互相影响 |
| 6. 领导力 | 通过影响力而非权威推进 |
| 7. 根据情境裁剪 | 没有通用方法，按需调整 |
| 8. 将质量融入流程 | 质量不是检测出来的，是做进去的 |
| 9. 驾驭复杂性 | 拥抱不确定，不要强求确定性 |
| 10. 优化风险应对 | 识别 + 分析 + 应对 + 监控 |
| 11. 拥抱适应性和弹性 | 按需调整，而非固守计划 |
| 12. 为未来创造变化 | 项目关闭后，价值要持续实现 |

### 2. 8 个绩效域

#### 1) 利益相关者（Stakeholders）
- 识别所有利益相关者（支持者、反对者、中立者）
- 分析他们的影响力和关注点
- 制定参与策略：告知 / 咨询 / 协作 / 赋权

#### 2) 团队（Team）
- 定义角色和职责（RACI 矩阵）
- 建立团队规范和协作方式
- FDE 场景：跨客户组织和自己公司的虚拟团队管理

#### 3) 开发方法与生命周期（Development Approach & Life Cycle）
- **预测型（瀑布）**：需求稳定，可以提前规划
- **适应型（敏捷）**：需求不确定，快速迭代
- **混合型**：按项目阶段选择适合的方法

AI 项目推荐：**混合型**——探索阶段用敏捷，部署阶段用预测

#### 4) 规划（Planning）
- 项目章程（Project Charter）：授权文件，定义目标和边界
- WBS（工作分解结构）：将交付物逐层分解到可管理的工作包
- 关键路径法（CPM）：识别无 Slack 的任务链，确定最短工期

#### 5) 项目工作（Project Work）
- 日常执行管理
- 问题日志（Issue Log）：记录、跟踪、解决
- 变更管理：变更请求 → 评估影响 → 批准/拒绝 → 执行

#### 6) 交付（Delivery）
- 范围管理：防止范围蔓延（Scope Creep）
- 质量管理：验收标准、测试、用户验收测试（UAT）
- 里程碑评审：每个阶段明确的"完成"定义（Definition of Done）

#### 7) 测量（Measurement）
**挣值管理（Earned Value Management，EVM）**
```
PV（计划价值）：按计划应该完成多少工作的预算
EV（挣值）：已完成工作的预算成本
AC（实际成本）：已完成工作的实际花费

进度偏差 SV = EV - PV（负数 = 落后）
成本偏差 CV = EV - AC（负数 = 超支）
进度绩效指数 SPI = EV/PV（<1 = 落后）
成本绩效指数 CPI = EV/AC（<1 = 超支）
```

#### 8) 不确定性（Uncertainty）
**风险管理流程**：
1. 识别风险：头脑风暴 + 历史经验
2. 定性分析：概率 × 影响矩阵
3. 定量分析：蒙特卡洛模拟（大项目用）
4. 应对策略：规避 / 转移 / 减轻 / 接受
5. 监控风险：定期审查，触发条件明确

### 3. AI 项目的 PMBOK 应用重点

#### AI 项目常见范围蔓延防控
- 明确定义 MVP 范围（数据范围、功能范围、用户范围）
- 变更请求需要评估对三角约束（范围/时间/成本）的影响
- "再加一个小功能"是最常见的范围蔓延来源

#### AI 项目的关键风险
| 风险类型 | 概率 | 影响 | 应对策略 |
|---------|------|------|---------|
| 数据质量不足 | 高 | 高 | 前期数据审计；数据准备冲刺 |
| 模型准确率不达标 | 中 | 高 | 设定可接受阈值；备用传统方案 |
| 用户采纳率低 | 高 | 高 | 变革管理计划；种子用户培育 |
| 第三方 API 变更 | 低 | 中 | 接口抽象层；替代方案 |
| 监管合规风险 | 中 | 高 | 法律顾问提前介入；合规检查清单 |

## 🔑 关键概念

### 范围蔓延（Scope Creep）
**定义**: 项目范围在未经正式变更控制的情况下逐渐扩大

**重要性**: AI 项目最常见的失败原因之一；客户总想"顺便加几个功能"

**应用场景**:
1. 每个新需求必须走变更请求流程，评估工期和成本影响
2. 项目章程中明确写出"范围外"的内容
3. 定期向客户确认当前范围，防止"口头承诺"积累

---

### RACI 矩阵
**定义**: 
- **R**esponsible：负责执行
- **A**ccountable：最终负责
- **C**onsulted：提供输入
- **I**nformed：需要被告知

**重要性**: 消除"谁来做这件事"的歧义

**应用场景**:
1. AI 项目：数据准备（R: 数据工程师，A: 数据负责人，C: 业务，I: 管理层）
2. 上线决策：（R: FDE，A: 客户 IT 总监，C: 业务，I: CEO）

## 💡 实战应用场景

### 场景 1: AI 项目启动（Project Charter）
**背景**: FDE 开始一个新客户的 AI 实施项目

**解决方案**: 用项目章程对齐所有利益相关者

**实施步骤**:
1. 明确项目目标：可量化的成功标准
2. 定义范围边界：In Scope（明确写出） vs Out of Scope
3. 确定约束条件：预算上限、时间节点、技术限制
4. 识别利益相关者：谁是赞助者？谁是终端用户？谁可能反对？
5. 获得书面签字：防止后续"我以为包括这个"

**预期效果**: 项目有清晰的边界，减少后续争议

---

### 场景 2: 项目延期时如何沟通
**背景**: AI 项目因数据质量问题延期 3 周

**解决方案**: 用 EVM + 变更请求流程透明沟通

**实施步骤**:
1. 准备数据：SPI < 1，说明当前进度
2. 分析根因：数据质量问题导致额外 X 天工期
3. 提出解决方案：接受延期 3 周 / 缩减范围保持时间 / 增加资源
4. 呈现风险：不同方案的利弊
5. 让客户做决策，FDE 提供建议

**预期效果**: 客户感到被尊重（知情权）而不是被惊吓
