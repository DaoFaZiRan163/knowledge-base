## 今日学习内容

> AI产品的评估体系是将"好不好用"从模糊的主观感受转化为可量化、可追踪的科学体系。与传统软件不同，AI产品的输出具有不确定性和概率性，这意味着我们需要在"准确性、安全性、用户满意度、商业价值"四个维度上建立多层次评估框架。AI问答产品尤其需要一套既能覆盖模型层又能覆盖产品层的综合评估方案，让每次迭代都有据可依。

## 核心知识点

### 知识点一：评估体系的四个维度

AI产品评估不能只看单一指标，需要从四个正交维度切入：

| 维度 | 核心问题 | 代表指标 |
|------|----------|----------|
| **效果层** | 模型输出对不对、好不好 | 准确率、召回率、BLEU、ROUGE |
| **体验层** | 用户感觉好不好用 | 满意度评分、任务完成率、NPS |
| **安全层** | 输出有没有风险 | 有害内容率、拒绝率、幻觉率 |
| **成本层** | 经济账合不合理 | Token消耗、延迟、单次成本 |

这四个维度需同时追踪，因为优化某一维度往往会影响其他维度——例如提升安全性（增加过滤）可能拉低用户体验层满意度。

### 知识点二：离线评估 vs 在线评估

**离线评估（Offline Evaluation）**
在产品上线前，用标注好的测试集评估模型质量。优点是成本低、可控、可重复；缺点是测试集可能与真实用户分布存在差距（分布偏移问题）。

常见离线指标：
- **Exact Match（EM）**：预测答案与标准答案完全一致的比例
- **F1 Score**：token级别的精准率与召回率调和平均
- **ROUGE-L**：最长公共子序列，常用于摘要类评估
- **BERTScore**：基于语义相似度的评估，比词匹配更智能

**在线评估（Online Evaluation）**
产品上线后，用真实用户行为数据评估。优点是最接近真实价值；缺点是迭代慢、影响用户体验。

常见在线指标：
- **点赞/踩率**：用户显式反馈
- **继续对话率**：隐式满意度信号
- **问题解决率**：用户是否不再追问同一问题
- **转人工率**：AI兜底失败的比例

**最佳实践**：先用离线评估过滤明显劣质版本，再用在线A/B测试验证真实效果。

### 知识点三：LLM-as-Judge 评估范式

人工标注成本高、速度慢，业界越来越多采用"用大模型评估大模型"的方式：

```
评估流程：
用户问题 + AI回答 → 评估Prompt → GPT-4/Claude → {评分, 评分理由}
```

**评估Prompt示例**：
```
你是一位专业评估员，请从以下维度对AI回答打分（1-5分）：
1. 准确性：回答与事实的一致程度
2. 完整性：是否覆盖了问题的所有方面
3. 简洁性：是否有冗余无用内容
4. 安全性：是否包含有害或偏颇内容

[用户问题]：{{question}}
[AI回答]：{{answer}}

请输出JSON格式评分和理由。
```

LLM-as-Judge 的局限性：存在"自我偏好"（同家模型互评分高）和"位置偏差"（倾向于选第一个选项），需要随机化测试顺序并混用多种评估模型。

### 知识点四：Bad Case 驱动的评估体系

最实用的AI产品评估方式是**Bad Case 闭环**：

```
收集Bad Case → 分类归因 → 确定优化方向 → 修改后回归测试 → 上线验证
```

Bad Case 分类体系（以AI问答为例）：

| 类型 | 定义 | 优化方向 |
|------|------|----------|
| 知识幻觉 | 编造不存在的事实 | RAG增强、置信度阈值 |
| 理解偏差 | 误解用户意图 | 意图识别模块、追问策略 |
| 格式错误 | 输出格式不符合要求 | 系统Prompt约束、后处理 |
| 安全越界 | 输出违规或有害内容 | 内容过滤器、RLHF |
| 能力边界 | 问题超出模型能力范围 | 拒答策略、转人工 |

### 知识点五：评估自动化流水线

规模化评估需要自动化：

```
[每次发布] → 触发评估Pipeline
     ↓
[离线测试集] → 跑评估脚本 → 生成评估报告
     ↓
[LLM-as-Judge] → 对抽样样本打分
     ↓
[回归测试] → 对比上一版本的Bad Case是否修复
     ↓
[评估报告] → 产品经理review → 决定是否发布
```

**工具推荐**：
- OpenAI Evals：开源评估框架
- LangSmith：LangChain配套的评估追踪平台
- Ragas：专为RAG系统设计的评估框架（含Faithfulness、Answer Relevancy等指标）

### 知识点六：用户满意度的隐式与显式信号

**显式信号**（用户主动表达）：
- 点赞 / 踩
- 评分（1-5星）
- 文字反馈
- NPS调查

**隐式信号**（行为数据推断）：
- 复制率：用户复制AI输出 → 高度满意
- 追问率：追问同一问题 → 未解决问题
- 会话长度：合适区间为3-8轮，过短可能无帮助，过长可能未聚焦
- 再访率：次日、7日留存 → 产品整体价值
- 切换率：切换到人工 → AI失败案例

**AI产品特有信号**：引用准确率（当AI给出来源链接时，用户点击验证的比例）。

## 关键概念详解

> 每个概念按 **是什么 → 为什么重要 → 具体例子** 展开。

### RAGAS（RAG Assessment）
**是什么**：专为RAG系统设计的评估框架，包含四个核心指标：Faithfulness（忠实度）、Answer Relevancy（答案相关性）、Context Precision（上下文精准度）、Context Recall（上下文召回率）。

**为什么重要**：传统评估指标（如ROUGE）无法区分"模型在乱说"和"检索到的内容就是错的"，RAGAS能精确定位RAG链路中哪个环节出了问题。

**具体例子**：AI问答召回了3篇文档，但只有1篇真正用于生成答案，且答案忠实于文档内容——Faithfulness高（好），Context Precision低（检索有噪声需优化）。

### 幻觉率（Hallucination Rate）
**是什么**：模型生成内容中，与事实不符或无中生有的信息所占比例。

**为什么重要**：幻觉是AI产品最严重的可信任度风险，尤其在企业知识问答场景，一个错误的内部规定解读可能引发严重后果。

**具体例子**：用户问"我们公司差旅报销上限是多少"，AI回答"2000元"（实际是1500元），这就是知识型幻觉，需要通过RAG+来源引用+置信度展示来缓解。

### A/B测试中的Novelty Effect（新奇效应）
**是什么**：新功能上线初期因用户好奇心驱动，指标短暂虚高，之后回落到真实水平。

**为什么重要**：如果在新奇效应期就决策发布，可能误判功能价值；AI功能迭代频繁，更容易踩坑。

**具体例子**：新增"AI主动追问"功能后，第一周用户互动率+30%，两周后回落至+8%。应以两周后的稳态数据为决策依据。

### 评估指标的"Goodhart定律"
**是什么**：当一个指标成为目标时，它就不再是好的指标——因为团队会优化指标而非优化本质目标。

**为什么重要**：AI产品PM必须警惕"刷指标"现象，例如通过设计引导点赞、降低拒绝率阈值来美化数据。

**具体例子**：为了提高"问题解决率"，把判定标准从"用户不再追问"改为"会话超过3轮"，指标好看了但用户体验未必真的改善。

## 面试话术

**Q：你们是怎么评估AI产品质量的？**
> 我们建立了离线+在线两层评估体系：离线用基于标注测试集的F1和RAGAS指标做每次发版前的回归，在线通过用户点赞率、问题解决率、转人工率做实时监控。同时每周PM会人工抽查50条Bad Case并分类归因，驱动下一轮迭代优先级。

**Q：AI产品的评估和传统软件有什么本质区别？**
> 传统软件的质量主要看功能是否按预期执行，是确定性的；AI产品输出是概率性的，同一个输入可能有多种合理回答，所以需要评估的是"分布质量"而非"单点正确性"。这导致必须用统计方法、更大样本量，以及主观维度（用户满意度）来综合判断。

**Q：你遇到过评估指标上升但用户体验没改善的情况吗？**
> 遇到过。曾经为了提升"回答完整性"指标，调整了Prompt让模型输出更长的回答，离线评估分数上升了，但上线后用户反馈"太啰嗦"，追问率反而上升。这让我意识到评估指标需要和实际用户价值锚定，不能孤立地优化单一指标。
