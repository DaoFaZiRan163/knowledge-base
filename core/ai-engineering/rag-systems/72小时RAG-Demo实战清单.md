# 72 小时为客户做出第一个可用的 RAG Demo —— 一份 FDE 的实战清单

## 🎯 核心定义

**72小时RAG Demo** 是 Forward Deployed Engineer 在客户现场交付的第一个可工作原型的标准动作。核心思想是：**Demo 是一个"赌注"，证明可能性而非完美产品**。目标是争取 4 周 pilot，而不是交付生产级软件。

> "3 天搞定一个 AI Demo" 听上去像销售话术。把它拆成 Day 0–3 四段清单后，会发现这其实是 Forward Deployed Engineer 的标准动作。

## 🗓️ 时间线框架

### Day 0: 入场前必确认的五个问题

在到达客户现场之前，必须澄清：

| 问题 | 核心目的 |
|------|---------|
| **业务问题** | 最痛苦的场景是什么？代价多大？ |
| **数据格式** | 在哪？量多大？更新频率？敏感度？ |
| **成功标准** | 如何判断"通过/失败"？ |
| **受众** | 技术还是业务干系人？ |
| **边界** | 安全/合规限制有哪些？ |

### Day 1: 数据 + 检索骨架

**Day 1 规则：写零个 prompt。** 全力聚焦检索质量，因为"90% 的 RAG 失败发生在检索阶段，而非生成阶段"。

**关键步骤：**
1. 数据清洗
2. 分块（300-500 tokens）
3. 嵌入（使用 bge-m3 或 text-embedding-3-small）
4. 测量 Recall@5 / Recall@10

**目标：** Recall@5 达到 70%+ 再进入下一步。若未达到：
- 添加元数据过滤器
- 混合 BM25 + 向量检索
- 查询改写

### Day 2: RAG Chain + Demo 集合

构建最小端到端 pipeline：`query → retrieval → context → LLM → response + citations`

**准备 8-12 个 Demo 问题：**
- 3 个明显成功案例
- 3 个边缘案例
- 2 个"应该承认无知"的案例
- 1-2 个真实客户场景

**演练：** 在 Day 3 前与客户对接人进行 2 小时彩排。

### Day 3: Demo + 诚实展示失败

**主动展示 1-2 个真实失败案例。**

> "见过精美 PoC 的客户最怕的是上线后才发现问题。展示'我们知道为什么这个会失败'能建立信任。"

**Demo 结构：**
| 时间 | 内容 |
|------|------|
| 5 min | 回顾 Day 0 目标 |
| 10 min | Live Demo 4-6 个问题 |
| 5 min | 指标（Recall@5、准确率、响应时间） |
| 10 min | 4 周 pilot 提案 |

## ⚠️ 三大常见陷阱

### 陷阱 1: 使用合成数据
客户能立即识别合成数据。永远使用真实数据。

### 陷阱 2: 把模型选择当优先级
**检索质量比模型选择更重要。** 先优化检索，再考虑模型。

### 陷阱 3: 美化 UI
客户关心的是"能不能回答问题"，不是界面好不好看。

## 📝 实战要点

### FDE 核心思维

1. **Demo 不是产品，是赌注** — 证明可能性，争取 pilot
2. **检索先于生成** — 90% 失败在检索，10% 在生成
3. **诚实建立信任** — 主动展示失败比伪装成功更有价值
4. **快速迭代** — 4 周 pilot > 3 个月生产开发

### 技术优先级

```
1. 检索质量（Recall@5 ≥ 70%）
2. 端到端可用性
3. 引用来源透明度
4. 响应速度
5. 界面简洁（非优先）
```

## 📊 质量指标

|------|--------|---------|
| Recall@5 | ≥ 70% | 检索结果相关性评估 |
