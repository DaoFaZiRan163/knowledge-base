# Prompt Engineering 的工程化：从个人技巧到团队规范

## 🎯 核心问题

> "早期 prompt 依赖个人调试，人员变动导致质量波动。"

Prompt 应被视为**产品规则、业务知识与模型约束的组合**，像代码一样被管理。

## 🏗️ 模板要素

| 要素 | 说明 | 示例 |
|------|------|------|
| **角色定义** | 定义 AI 身份 | "你是一个客服助手"（避免过多人格堆叠） |
| **明确目标** | 清晰的任务描述 | "回答用户关于订单状态的问题" |
| **结构化输入/输出** | 明确的格式要求 | JSON / Markdown / 纯文本 |
| **约束条件** | 禁止行为 | "不知道时回答'暂不了解'" |
| **边界处理** | 异常情况 | "涉及退款转人工" |
| **示例** | Few-shot 学习 | 提供 2-3 个正确示例 |

## 📊 三级版本管理

| 类型 | 用途 | 变更频率 |
|------|------|---------|
| **系统级** | 安全边界 | 低 |
| **任务级** | 执行方式 | 中 |
| **场景级** | 业务适配 | 高 |

## 🔄 评审机制

```
样本回归 → 同伴评审 → 灰度发布 → 回滚方案
```

## 🎯 评估器作用

**上线前验证：**
- 事实一致性
- 格式正确性
- 证据引用
- 转人工判断

## 💡 实践建议

> "能用结构化输入解决的，不靠自然语言提醒；能用评估器兜底的，不指望模型永远自觉。"

## 📋 落地要素

| 要素 | 说明 |
|------|------|
| **命名规则** | 统一的命名规范 |
| **模板结构** | 清晰的模板格式 |
| **变更流程** | 版本控制和审批 |
| **评估样本** | 基准测试集 |
| **发布节奏** | 灰度策略 |
| **日志字段** | 追踪字段 |

## 📈 核心结论

> "工程化让团队从'会调 Prompt'变成'能管理 Prompt'，前者靠经验，后者靠体系。"
