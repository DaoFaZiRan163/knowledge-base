## 今日学习内容

> 从今天起进入 Phase 2，深入 AI 技术的内部原理。作为 AI PM，你不需要能自己训练一个 LLM，但你必须理解它是"如何工作的"，以及这些工作原理对产品决策意味着什么。LLM（大语言模型）本质上是一个学会了"预测下一个词"的超大规模统计系统——理解这一点，你就能明白为什么 AI 会"幻觉"、为什么 Prompt 设计如此重要、为什么上下文窗口大小影响产品设计。这节课是整个 AI 技术认知阶段的基石。

## 核心知识点

### 知识点一：LLM 的本质——"预测下一个词"

LLM 的核心任务极其简单（但实现极其复杂）：**给定前面的文字，预测下一个最可能出现的词（Token）**。

```
输入：  "今天天气很__"
模型：  计算所有词出现在这里的概率
输出：  "好"（概率0.45）、"糟"（0.20）、"热"（0.15）、...
```

这个看似简单的任务，当训练数据足够大（TB级别的人类文字）、模型参数足够多（数百亿到数万亿）时，模型会"自动学会"：语法规则、事实知识、推理逻辑、代码编写、甚至人类的说话风格。

**对产品的启示**：LLM 的输出是**概率性的**，同一个输入每次可能产生不同输出（Temperature 参数控制随机程度）。这意味着你无法用"确定性测试"来验收 AI 功能——需要统计意义上的质量评估。

### 知识点二：Transformer 架构核心概念

现代 LLM 几乎全部基于 Transformer 架构，你需要理解以下几个关键概念（不需要会写代码）：

**Token（词元）**：
- 文本被切分成 Token，不等于一个字/词
- 英文大约4个字符=1个Token，中文约1-2个汉字=1个Token
- GPT-4 的上下文窗口为128K Tokens ≈ 约20万中文字符
- **产品意义**：长对话、长文档处理受 Token 限制约束，超出会截断

**Attention（注意力机制）**：
- 模型在生成每个词时，会"关注"输入中哪些部分与当前位置最相关
- 使 LLM 能处理长距离依赖关系（例如理解段落之间的逻辑）
- **产品意义**：上下文越长，推理成本越高（Attention 计算量是 O(n²)）

**自回归生成**：
- LLM 每次只生成一个 Token，然后将已生成内容加入输入，继续生成下一个
- 这就是为什么流式输出（Streaming）是一个词一个词蹦出来，而非等待完整结果
- **产品意义**：流式输出显著改善用户等待体验，是 AI 问答产品的必要设计

### 知识点三：LLM 的训练过程（三阶段理解）

```
阶段1：预训练（Pre-training）
  └── 输入：互联网海量文本（Common Crawl、Books、Wikipedia等）
  └── 任务：预测下一个词
  └── 产出：具备语言理解和知识储存的基础模型（Base Model）
  └── 成本：数百万至数亿美元，需数千张GPU卡数月训练

阶段2：指令微调（Instruction Fine-tuning / SFT）
  └── 输入：高质量的"指令-回答"对（人工标注）
  └── 任务：学会按指令执行，而不只是补全文字
  └── 产出：能对话、能执行任务的 Chat 模型
  └── 成本：远低于预训练，可在数天内完成

阶段3：强化学习对齐（RLHF）
  └── 输入：人类对模型输出的偏好排序（哪个回答更好？）
  └── 任务：训练模型输出人类偏好的内容，避免有害输出
  └── 产出：更安全、更有用、更符合人类价值观的模型
  └── 成本：中等，关键在于高质量的人类反馈数据
```

**对产品的启示**：
- 用你自己的数据做"指令微调"可以显著提升领域专业度（参见 Day 58）
- 用户的差评数据可以转化为 RLHF 训练素材，形成良性数据飞轮

### 知识点四：LLM 的核心能力与局限性

**LLM 擅长的事**：

| 能力类别 | 具体表现 | AI问答应用 |
|---|---|---|
| 文本生成 | 写作、摘要、翻译、改写 | 生成回答、改写问题 |
| 信息提取 | 从文本中提取关键信息 | 从文档中找答案（RAG基础）|
| 推理 | 逐步分析、解决逻辑问题 | 复杂问题拆解分析 |
| 代码 | 写代码、解释代码、Debug | 技术类问答 |
| 对话 | 多轮上下文理解 | 追问、澄清、引导 |

**LLM 的本质局限**：

| 局限 | 原因 | 产品应对方案 |
|---|---|---|
| 幻觉（Hallucination）| 预测下一个词，不是"查询真相" | RAG接地气（Day 19）|
| 知识截止日期 | 训练数据有时效性 | 搜索增强、实时数据注入 |
| 无法访问实时信息 | 不联网，不知道今天发生了什么 | Function Calling + 搜索API |
| 长上下文遗忘 | Attention 在极长上下文中退化 | 分段处理、记忆压缩 |
| 推理不稳定 | 复杂多步推理准确率下降 | Chain-of-Thought Prompt（Day 17）|
| 无状态 | 每次对话独立，无持久记忆 | 显式记忆管理（数据库存储）|

### 知识点五：LLM 的关键参数——产品决策影响

**Temperature（温度）**：
- 控制输出随机性，0=完全确定性，1=正常随机，2=非常随机
- 产品应用：创意写作用高温度（0.8-1.0），精确问答用低温度（0.1-0.3）

**Context Window（上下文窗口）**：
- 模型一次能处理的最大 Token 数
- 直接影响：对话能记住多少轮、文档能处理多长
- 产品决策：长对话记忆、长文档分析功能的技术可行性

**Max Tokens（最大输出长度）**：
- 控制单次回答的最大长度
- 过长：成本高、用户阅读负担重；过短：回答不完整
- 建议：不同场景设置不同上限（简单问答500 Token，分析报告3000 Token）

## 关键概念详解

### Hallucination（幻觉）
**是什么**：LLM 以高置信度生成看似正确但实际上错误或无中生有的内容。称为"幻觉"是因为模型对错误内容没有任何"怀疑"——它就像在做梦，梦中的内容感觉非常真实。

**为什么重要**：幻觉是 AI 问答产品信任危机的根源。用户一旦发现 AI 自信地给出了错误信息，对整个系统的信任度会大幅下降。对于医疗、法律、金融等高风险领域，幻觉可能造成实质伤害。

**具体例子**：用户问"请介绍一下张伟教授在清华大学的研究成果"，AI 可能会生成一个详细且流畅的"张伟教授"介绍，包含具体的论文标题、研究方向和获奖情况——但这些内容可能完全是编造的，因为 AI 无法区分"知道某人"和"凭感觉生成关于某人的内容"。

---

### Token 与上下文窗口
**是什么**：Token 是 LLM 处理文本的基本单位，大约等于一个词或一个汉字。上下文窗口是模型一次能"看到"和"记住"的最大 Token 数，超出窗口的内容会被截断，模型无法看到。

**为什么重要**：上下文窗口直接决定你的产品能做什么——能支持多长的文档分析、多少轮的对话历史、多复杂的系统提示词。对 AI 问答产品，合理管理上下文窗口是控制成本和保证质量的关键工程能力。

**具体例子**：你的 AI 问答产品使用 GPT-4（128K Token 上下文）。一个用户上传了一份10万字的产品文档（约65K Token），加上系统提示词（5K Token）和对话历史（10K Token），总计80K Token，在窗口内——OK。但如果文档超过100页，就需要 RAG 技术来分段处理（Day 19详解）。

---

### 预训练知识 vs 上下文知识
**是什么**：预训练知识是 LLM 从训练数据中"学到"并存储在参数中的知识，是固化的；上下文知识是你在 Prompt 中实时提供给模型的信息，每次对话独立。

**为什么重要**：理解这两类知识的区别是 AI 产品架构设计的基础。预训练知识有截止日期、可能有偏差；上下文知识灵活但受窗口限制。好的 AI 产品设计是让这两者协同工作（RAG 就是最典型的做法）。

**具体例子**：你问 AI "我们公司2025年的新产品是什么？"——这是上下文知识，训练数据中没有，AI 无法回答或会幻觉。但如果你在 Prompt 中提供了"公司产品手册.pdf"的关键内容，模型就能基于上下文知识准确回答。

## 面试话术

**Q：用白话解释一下什么是大语言模型（LLM）。**
> 大语言模型本质上是一个通过阅读了海量人类文字而变得"无比博学"的预测系统。它每次做的事情很简单：给定前面的文字，猜测下一个最可能出现的词。但当这个系统的规模足够大、训练数据足够多时，它就"涌现"出了理解、推理、写作、甚至编程的能力——就像一个读了几亿本书的人，问什么都能回答。当然，它的局限也来自于此：它不是在"查真相"，而是在"做最合理的猜测"，所以会出现看似很自信但实际错误的"幻觉"。

**Q：LLM 为什么会产生幻觉？作为 PM 你如何应对？**
> 幻觉的根本原因是 LLM 的工作机制——它是在预测"最可能出现的下一个词"，而不是在查询一个真实可验证的知识库。当它不知道答案时，它会"编造"一个听起来合理的内容，因为这符合语言概率分布。作为 PM，我从三个维度应对：第一，产品层面用 RAG 技术为模型提供可靠的事实来源；第二，让模型在不确定时明确说"我不确定"而非自信生成（通过 Prompt 设计）；第三，建立 Bad Case 监控，对幻觉类错误单独追踪和处理。

**Q：上下文窗口对 AI 产品设计有什么影响？**
> 上下文窗口决定了很多产品能力的边界。比如，128K Token 的 GPT-4 理论上能处理约10万字的文档，但实际上在极长上下文中，模型的关注质量会下降（"Lost in the Middle"问题）。对 AI 问答产品，我需要在产品设计时考虑：如何管理多轮对话的历史（压缩、摘要还是截断？）、文档处理用分段 RAG 还是全文放入上下文？这些决策直接影响回答质量和单次调用成本。
