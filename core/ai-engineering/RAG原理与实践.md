## 今日学习内容

> RAG（Retrieval-Augmented Generation，检索增强生成）是当前AI知识库产品最核心的技术架构。它解决了大语言模型最致命的两个问题：知识截止日期限制和幻觉。对AI问答产品来说，RAG不是可选项，而是基础设施。理解RAG的工作原理和关键决策点，是PM与技术团队高质量沟通的前提，也是产品体验优化的核心抓手。

## 核心知识点

### 知识点一：为什么需要RAG——问题起点

不用RAG的AI问答产品面临的三大核心问题：

| 问题 | 表现 | 用户感知 |
|------|------|----------|
| 知识截止 | 模型不知道公司最新政策/产品文档 | "AI说的已经过时了" |
| 幻觉风险 | 模型编造不存在的政策条款/数据 | "AI在瞎说" |
| 无法定制 | 模型不了解公司特有的业务逻辑 | "AI不懂我们的业务" |

RAG的本质是：**给模型一个"考试小抄"** —— 在用户提问时，先从知识库里找出最相关的内容片段，塞给模型，模型再根据这些内容组织回答，而不是靠自己"背"出答案。

### 知识点二：RAG 完整工作流程

```
┌─────────────────────────────────────────────────────────┐
│                    离线索引阶段（一次性）                  │
│                                                         │
│  原始文档 → 文档解析 → 文本分块 → Embedding编码 → 向量存储  │
│  (PDF/Word/网页)  (提取纯文本)  (切成小段)  (转为向量)  (向量DB) │
└─────────────────────────────────────────────────────────┘
                         ↓（知识库建好后）
┌─────────────────────────────────────────────────────────┐
│                    在线查询阶段（每次对话）                  │
│                                                         │
│  用户问题 → 问题Embedding → 向量检索 → 相关文档片段          │
│     ↓                                      ↓           │
│  ┌──────────────────────────────────────────┐          │
│  │  组装Prompt：系统提示 + 检索内容 + 用户问题 │          │
│  └──────────────────────────────────────────┘          │
│                       ↓                                 │
│                    LLM生成回答                           │
│                       ↓                                 │
│              回答 + 来源引用 → 展示给用户                  │
└─────────────────────────────────────────────────────────┘
```

### 知识点三：RAG 关键决策点——产品影响最大的技术选择

**① 文本分块策略（Chunking）**

| 分块方式 | 优点 | 缺点 | 适用场景 |
|----------|------|------|----------|
| 固定长度（如每500字） | 简单均匀 | 可能切断语义单元 | 通用场景 |
| 按语义段落 | 保持语义完整 | 块大小不均 | 结构化文档 |
| 滑动窗口（有重叠） | 减少边界遗漏 | 存储量更大 | 重要信息在段落边界时 |
| 层级分块 | 支持粗粒度+细粒度 | 复杂度高 | 长文档、多层次查询 |

**PM关注点**：分块策略直接影响检索质量。如果用户问题跨越两个文本块，可能导致召回不完整，体现为"模型明明知道但没答出来"的Bad Case。

**② 检索方式**

```
向量检索（语义搜索）
  优点：理解语义，找到意思相近的内容
  缺点：精确词匹配可能不如关键字搜索
  
关键字检索（BM25等）
  优点：精确匹配，不会遗漏包含关键词的段落
  缺点：不理解语义，同义词会被遗漏

混合检索（Hybrid Search）= 向量 + 关键字
  最佳实践：两者结合，用RRF（倒数排名融合）算法合并结果
  大部分企业知识库产品推荐使用混合检索
```

**③ 召回数量（Top-K）**

- 召回太少（K=1-2）：信息不足，回答可能残缺
- 召回太多（K=10+）：噪音增加，超出Context Window，成本上升
- **推荐实践**：K=3-5作为初始值，根据Bad Case调整

### 知识点四：RAG 常见失效模式及解决方案

作为PM，你需要理解RAG失效时该如何诊断：

| 失效现象 | 根本原因 | 诊断方法 | 解决方向 |
|----------|----------|----------|----------|
| 明明有文档但没答出来 | 检索未召回相关文档 | 查看召回的Top-K文档是否包含答案 | 优化分块/检索方式 |
| 答案正确但来源错误 | 模型生成时混用了知识 | 核查引用内容与文档原文 | 加强Grounding指令 |
| 回答过时 | 知识库未更新 | 对比文档更新时间 | 建立文档更新机制 |
| 答案拼凑感强 | 召回文档碎片化 | 查看召回内容的连贯性 | 调整分块策略，增加块重叠 |
| 回答太长太碎 | LLM没有合理整合 | Prompt中未规定整合方式 | 优化Prompt的输出格式指令 |

### 知识点五：RAG 进阶——提升产品质量的高阶技术

**① 查询改写（Query Rewriting）**：
用户输入往往不是检索的最优形式，先用LLM改写用户问题，再去检索。例如"上次说的那个功能"→改写为"AI问答产品的记忆管理功能"，大幅提升召回率。

**② 重排序（Reranking）**：
初次检索后，用更精准的模型对Top-K结果重新排序，把最相关的放前面。常见方案：BGE Reranker、Cohere Rerank。

**③ 多路召回融合**：
对同一个问题，用多个不同的检索策略（向量/关键字/关系图谱）各自召回，再合并去重，提升召回全面性。

**④ 引用追踪**：
在产品层面展示"此答案来源于[文档名]第[X]节"，让用户可以核实，同时提升产品可信度。

## 关键概念详解

### Embedding（向量化）

**是什么**：将文本转化为高维数值向量的过程。语义相似的文本，其向量在空间中距离更近。这是向量检索的数学基础。

**为什么重要**：Embedding模型的质量直接决定了检索质量——模型理解"公司差旅报销流程"和"员工出差费用申请"是同一件事，才能在用户问其中一个时，召回与另一个相关的文档。

**具体例子**：在AI问答产品中，如果Embedding模型对中文理解不好（比如使用了英文为主的通用模型），会导致中文语义检索准确率低，用户问"如何请假"却召回了"如何请示领导"相关内容。推荐使用针对中文优化的Embedding模型（如BGE、M3E系列）。

---

### 向量数据库（Vector Database）

**是什么**：专门用于存储和检索向量的数据库，支持高效的相似度搜索。常见产品：Milvus、Weaviate、Pinecone、Chroma、PGVector（PostgreSQL插件）。

**为什么重要**：向量数据库的性能决定了检索速度，直接影响用户感知到的响应时间。对于大规模知识库（百万级文档片段），普通数据库的全量比较无法满足实时需求。

**具体例子**：如果你的AI问答产品有10万个文档片段，每次用户提问需要和这10万个片段比较相似度，如果用暴力搜索需要几百毫秒，向量数据库通过ANN（近似最近邻）算法可以在10ms内完成，用户无感知等待。

---

### 知识库更新机制

**是什么**：当企业文档更新时，知识库的索引需要同步更新的机制。包括：全量重建（慢但保证一致性）、增量更新（快但需要处理版本冲突）、实时同步（最新但成本最高）。

**为什么重要**：知识库更新滞后是企业AI产品最常见的投诉原因之一。用户看到AI给出过时信息时，会立刻失去信任。

**具体例子**：产品设计要点：①在文档管理系统中触发更新时，自动推送增量索引任务；②在回答中展示文档最后更新时间；③对用户举报"信息过时"的Bad Case建立快速响应通道，支持24小时内人工触发重建索引。

## 面试话术

**Q：请解释RAG是什么，以及为什么你们的AI产品使用它？**
> RAG是检索增强生成，核心思路是"先找再答"——用户提问时，先从知识库里检索最相关的文档片段，再把这些片段作为上下文交给LLM组织回答。我们使用RAG主要解决两个问题：一是知识截止问题，公司内部文档模型不可能提前训练，必须在查询时提供；二是幻觉问题，有了原始文档做"小抄"，模型不需要凭空生成，大幅降低了错误率，同时我们可以展示来源让用户验证。

**Q：你们RAG的效果怎么样？遇到过哪些问题？**
> 我们RAG上线后，有文档支撑的问题回答准确率从60%提升到了85%。遇到的主要问题有两个：一是"有答案但没召回"——通常是分块策略不合理导致答案被切断，我们通过增加块重叠率和引入混合检索解决了大部分；二是"答案正确但来源乱指"——这是LLM在整合多段文档时混淆了来源，我们在Prompt中加了强引用约束后改善明显。

**Q：RAG的哪个环节对最终效果影响最大？**
> 从我的经验来看，Embedding模型的选择和分块策略对效果影响最大，这两个决定了"能不能找到对的内容"，是RAG的地基。如果地基没打好，后面Reranking再精准也没用。排第三位的是Prompt中的Grounding指令，它决定了"找到内容后能不能正确使用"。
