## 今日学习内容

> 对AI产品经理而言，理解成本结构不只是财务知识，更是产品决策的底层逻辑。一个看似完美的AI功能，可能因为Token消耗过高而无法商业化；一个巧妙的Prompt压缩方案，可能让单次成本下降60%。本章从API计费原理出发，帮助你建立AI产品的成本心智模型，掌握成本优化的核心杠杆，并能向商业决策者清晰呈现AI产品的经济账。

## 核心知识点

### 知识点一：LLM API的计费原理

主流LLM API按**Token数量**计费，理解Token是成本计算的起点。

**什么是Token**：
- 1个英文单词 ≈ 1.3 个Token
- 1个中文字符 ≈ 1.5–2 个Token
- 1000个汉字 ≈ 1500–2000 个Token

**计费结构**（以主流大模型为例）：

| 模型类型 | 输入Token单价 | 输出Token单价 | 特点 |
|----------|--------------|--------------|------|
| 顶级大模型（Claude Opus/GPT-4o） | ¥0.10–0.15/千Token | ¥0.30–0.45/千Token | 能力强，成本高 |
| 中型模型（GPT-4o-mini等） | ¥0.01–0.03/千Token | ¥0.03–0.10/千Token | 性价比高 |
| 国产模型（文心/通义/豆包） | ¥0.004–0.02/千Token | ¥0.012–0.06/千Token | 中文优化，低价 |

**关键认知**：输出Token通常比输入Token贵2–4倍，因此控制输出长度是最直接的降本手段。

### 知识点二：一次AI问答的完整成本构成

一次完整的AI问答请求，Token消耗来自多个部分：

```
总Token = 系统提示(System Prompt) 
        + 历史对话(History) 
        + 用户输入(User Input) 
        + RAG检索内容(Context) 
        + AI输出(Output)
```

**典型Token分布（以企业知识问答为例）**：

| 组成部分 | 典型Token数 | 占比 | 优化空间 |
|----------|-------------|------|----------|
| 系统提示 | 300–800 | 15–30% | 压缩简化 |
| 历史对话 | 500–2000 | 20–40% | 滑动窗口 |
| RAG上下文 | 1000–3000 | 30–50% | 精准召回 |
| 用户输入 | 50–500 | 5–15% | 不可控 |
| AI输出 | 200–800 | 10–25% | 限制长度 |
| **合计** | **~2000–7000** | 100% | |

按中型模型¥0.02/千Token估算，单次问答成本 = 约 **¥0.04–0.14**。

### 知识点三：成本的五大优化杠杆

**杠杆一：模型路由（Model Routing）**
不同复杂度的问题用不同级别的模型处理：
- 简单问答（"你们几点开门"）→ 小模型或规则引擎
- 中等复杂（"帮我分析这份合同"）→ 中型模型
- 复杂推理（"给我写一份战略分析"）→ 大模型

**杠杆二：提示词压缩（Prompt Compression）**
- 删除不必要的示例和解释
- 将冗长系统提示提炼为简洁指令
- 使用Prompt缓存（Prompt Caching）：将不变的系统提示缓存，只为变化部分付费

**杠杆三：RAG精度提升**
召回更精准的文档块，减少噪声文档：
- 目标：召回的top-3文档中，有效文档占比 ≥ 85%
- 从召回5段改为召回3段，直接减少30–40% RAG相关Token

**杠杆四：对话历史压缩**
- 滑动窗口：只保留最近N轮对话（而非全量）
- 摘要压缩：将早期对话总结为摘要，大幅压缩历史Token
- 话题切换检测：新话题时清空历史，避免无关上下文

**杠杆五：输出长度控制**
- 在系统提示中明确要求"简洁回答，不超过200字"
- 用 `max_tokens` 参数限制输出上限
- 对标准化问题（FAQ类）使用缓存直接返回

### 知识点四：成本与质量的权衡矩阵

成本优化不能无限压缩，必须找到成本与质量的平衡点：

| 优化手段 | 降本效果 | 质量影响 | 推荐场景 |
|----------|----------|----------|----------|
| 换用小模型 | 70–90% | 中等（视任务） | 简单FAQ、分类任务 |
| 压缩系统提示 | 10–20% | 低 | 通用场景 |
| 减少RAG召回数 | 20–40% | 中等 | 知识库覆盖率高时 |
| 滑动窗口历史 | 20–50% | 低（短对话） | 多轮对话 |
| 限制输出长度 | 20–40% | 低（简单问题） | 标准化问答 |
| Prompt缓存 | 30–70% | 无 | 系统提示稳定 |

### 知识点五：AI产品的完整成本结构（超出API调用）

API调用费用只是AI产品成本的一部分，完整成本结构包括：

```
AI产品成本 = 
  模型调用费（API Token费）
  + 向量数据库费（存储+查询）
  + 计算基础设施费（服务器、CDN）
  + 数据处理费（文档解析、清洗）
  + 人工审核费（Bad Case review）
  + 知识库维护费（内容更新、标注）
  + 安全合规费（过滤器、审计）
```

对大多数中型AI问答产品，API调用费占总AI相关成本的 **40–60%**，其余是支撑性基础设施。

### 知识点六：从成本结构到定价策略

理解成本是制定定价的前提：

**成本导向定价**：
- 单次会话成本 × 安全系数（1.5–3x）= 最低定价单元
- 例：单次成本¥0.08，目标毛利率60% → 定价 ¥0.20/次

**价值导向定价**（更推荐）：
- 分析AI替代了什么人工成本
- 例：一个问题原来需要客服1分钟处理，客服时薪¥30 → AI问答的价值上限是¥0.50/次

**常见定价模型**：

| 模式 | 适合场景 | 风险 |
|------|----------|------|
| 按问题数量计费 | 用量可预测 | 用户追求少问 |
| 按月订阅（问题配额） | B端企业客户 | 重度用户亏损 |
| 按使用时长 | 工具类产品 | 低效使用难区分 |
| Freemium（免费+付费） | C端获客 | 免费用户成本高 |

## 关键概念详解

> 每个概念按 **是什么 → 为什么重要 → 具体例子** 展开。

### Token经济学（Token Economics）
**是什么**：研究AI系统中Token的消耗、成本与价值关系的分析框架，帮助PM做出成本与质量的最优权衡决策。

**为什么重要**：在AI功能设计阶段就考虑Token经济学，可以避免上线后因成本失控而被迫降级功能或紧急优化的被动局面。

**具体例子**：产品设计时允许用户上传10页PDF进行问答，但未估算到每次检索会召回全文（约15000 Token），导致单次成本达¥0.80，是预算的8倍，不得不上线后紧急压缩至最多3页。

### Prompt缓存（Prompt Caching）
**是什么**：将频繁使用的长系统提示在API层缓存，重复请求只为不缓存的部分付费，大幅降低成本。

**为什么重要**：系统提示往往占总Token的20–40%，且每次请求都要重复发送，Prompt缓存是无损降本的最优手段。

**具体例子**：系统提示包含公司规则文档（2000 Token），每天被调用10万次。若Prompt缓存命中率90%，则每天节省 2000 × 90% × 10万 = 1.8亿 Token，以¥0.01/千Token算，每天节省¥1800。

### 毛利率与单位经济模型
**是什么**：AI产品的单位经济模型描述每产生一个单位收入（如一次会话/一个用户月）所对应的成本与收益，即 LTV/CAC 和 Gross Margin 分析。

**为什么重要**：许多AI产品看起来收入增长很快，但如果单位经济不健康（如毛利率为负），规模越大亏损越多，无法持续。

**具体例子**：月费¥199的AI问答SaaS，重度用户每月发送500条问题，平均每条成本¥0.20，光AI成本就达¥100/人/月，叠加其他成本后毛利率仅20%，说明定价需要上调或需要更精细的用量分级。

## 练习任务输出

**题目**：计算AI问答的单次回答成本模型

### 参考计算模板

**假设条件**（请根据你的实际情况填写）：

| 参数 | 示例值 | 你的值 |
|------|--------|--------|
| 使用的模型 | GPT-4o-mini | |
| 输入Token单价 | ¥0.015/千Token | |
| 输出Token单价 | ¥0.06/千Token | |
| 系统提示Token数 | 500 | |
| 平均RAG召回Token数 | 1500 | |
| 平均历史对话Token数 | 800 | |
| 平均用户输入Token数 | 150 | |
| 平均AI输出Token数 | 400 | |

**成本计算**：

```
输入Token合计 = 系统提示 + RAG + 历史 + 用户输入
             = 500 + 1500 + 800 + 150 = 2950 Token

输入成本 = 2950 × ¥0.015 / 1000 = ¥0.044
输出成本 = 400 × ¥0.060 / 1000 = ¥0.024
单次API成本 = ¥0.068

向量检索成本（Pinecone等）≈ ¥0.005/次
其他基础设施分摊 ≈ ¥0.005/次

单次问答总成本 ≈ ¥0.078
```

**月度成本估算**（假设1000 DAU，每人每天3次问答）：
```
月问答量 = 1000 × 3 × 30 = 90,000 次
月AI成本 = 90,000 × ¥0.078 = ¥7,020/月
```

## 面试话术

**Q：你们的AI产品成本结构是怎样的？如何控制成本？**
> 我们的成本主要由三块构成：LLM API调用费（约55%）、向量数据库检索费（约20%）和基础设施费（约25%）。控制成本的核心手段有三个：一是模型路由，简单FAQ走小模型节省70%以上；二是RAG精度优化，将召回文档块从5个压缩到3个，且只召回高相关段落；三是对话历史用滑动窗口，只保留最近5轮。综合下来我们的单次会话成本比初期下降了约45%。

**Q：如果DAU增长10倍，你的成本会同比增长10倍吗？**
> 不一定。边际成本方面，API调用费确实几乎线性增长；但固定成本（团队、基础设施基础费）被摊薄，向量数据库在一定规模内边际成本递减。更重要的是，规模增大后可以谈到更好的API折扣，以及有更多Bad Case数据来做模型优化，让小模型承接更多请求，推动单次成本下降。所以总成本增速理论上应低于DAU增速。
