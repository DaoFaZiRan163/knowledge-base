# LangGraph：用状态机思维重构 AI 问答系统的工作流编排

## 🎯 核心问题

**Linear Chain 的局限性：**
> "Linear Chain 假设每一步都会成功，但真实 AI 应用充满条件判断、分支跳转与重试循环。"

Linear Chain 在以下场景表现不佳：
- 复杂多意图查询（同时包含订单查询、换货申请、积分说明）
- 自动重试机制
- 状态回溯
- 多 Agent 协作

## 🏗️ LangGraph vs LangChain Chain 对比

| 能力维度 | LangChain Chain | LangGraph |
|---------|-----------------|-----------|
| 流程结构 | 线性顺序 | 有向图，支持分支/并行/循环 |
| 状态管理 | 无持久化状态 | 全局 State 对象 |
| 动态路由 | 不支持 | 条件边实现 |
| 重试/循环 | 需手写外层循环 | 图原生支持 |
| 多 Agent 协作 | 需大量胶水代码 | 子图嵌套天然支持 |
| Human-in-the-loop | 基本不支持 | Checkpoint 机制 |

## 🔑 三大核心概念

### 1. State（状态）

> "整个图共享一个全局状态对象，每个节点读取状态、执行任务、写回状态，驱动整条工作流向前推进。"

**State 包含字段：**
```python
{
    "user_input": str,           # 用户原始输入
    "intent_result": dict,       # 意图识别结果
    "tool_calls": list,          # 已调用工具列表
    "messages": list,           # 历史消息
    "answer_draft": str,        # 答案草稿
    "eval_score": float,        # 评估分数
    "retry_count": int           # 重试计数器
}
```

### 2. Node（节点）

> "Node 是纯函数：接收当前 State，返回更新后的 State。"

节点可执行：
- LLM 调用
- 数据库查询
- API 调用
- 启动嵌套子图

### 3. Edge（边）

> "条件边接受一个路由函数，该函数读取当前 State，返回下一个节点名称。"

**评估器重试逻辑：**
| 分数区间 | 动作 |
|---------|------|
| ≥ 75 分 | 跳到 END，直接输出 |
| < 60 分 | 回跳生成节点重试 |
| 60-74 分 | 触发人工审核 |

## 📊 工作流架构

```
用户输入
    ↓
意图识别节点
    ↓
策略路由条件边
    ↓
┌─────────────────────────────────────┐
│  三层策略选择                        │
│  ├── FAQ 精确匹配（<100ms）          │
│  ├── 关键词 + RAG 检索               │
│  └── Agent 编排（复杂意图）           │
└─────────────────────────────────────┘
    ↓
答案生成节点
    ↓
质量评估条件边
    ↓
输出
```

**关键设计决策：**
> "能不用 LLM 就不用 LLM"——精确匹配 <100ms 优先，节省成本降低延迟。

## 📈 落地效果

- 无效 LLM 调用减少约 **40%**
- 排查问题从"看日志猜路径"变为"直接回放 State 时间线"

## ⚠️ 踩过的坑

### 1. State 字段膨胀
**问题：** State 对象变得臃肿难以维护

**解决：** 引入分组命名约定
```
intent_*    # 意图相关
eval_*      # 评估相关
agent_*     # Agent 相关
```

### 2. 子图 State 继承问题
**问题：** 嵌套子图无法访问父图 State

**解决：** 显式做字段映射

### 3. 条件边调试难度
**问题：** 条件分支难以追踪

**解决：** 在路由函数加结构化日志，接入链路追踪系统

## 💡 经验总结

> "节点粒度要恰当——以'一个节点只做一件能被单独测试的事'为原则，但把高度耦合的子步骤放在同一个节点里。"

**判断标准：** 如果 AI 工作流需要以下任一特性，应考虑 LangGraph：
- 动态路由
- 循环重试
- 多 Agent 协作
- Human-in-the-loop
- 超过 3 个有状态依赖的步骤
