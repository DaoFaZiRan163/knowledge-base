# 《设计数据密集型应用》- Martin Kleppmann（DDIA）

## 📖 基本信息
- **作者**: Martin Kleppmann
- **出版年份**: 2017
- **核心主题**: 现代分布式数据系统的原理、权衡与实践——从数据模型到分布式事务
- **目标读者**: 希望深入理解数据系统底层机制的高级工程师和架构师

## 📚 知识框架

### 主要章节/模块

### Part I: 数据系统的基础

### 1. 可靠性、可扩展性、可维护性
- **可靠性**：出错时仍能正常工作（容错，而非无错）
- **可扩展性**：负载增长的应对能力；Twitter 推文扇出问题案例
- **可维护性**：可操作性 / 简洁性 / 可演化性
- **性能描述**：用百分位数（p99）而非平均值衡量延迟

### 2. 数据模型与查询语言
- **关系模型**：声明式查询，查询优化器自由选择执行计划
- **文档模型**：局部性好（一次读取整个文档），模式灵活；不擅长 Join
- **图模型**：适合多对多关系复杂的场景（社交图谱、知识图谱）
- **Schema-on-write vs Schema-on-read**：强类型 vs 读时解析

### 3. 存储与检索
- **LSM Tree（Log-Structured Merge Tree）**
  - 写入先进内存（MemTable），刷盘为不可变 SSTable
  - 后台 Compaction 合并，写放大换取写入性能
  - 用于：Cassandra、LevelDB、RocksDB、HBase
- **B-Tree**
  - 就地更新，读性能好，适合 OLTP
  - WAL（Write-Ahead Log）保证崩溃恢复
  - 用于：几乎所有关系型数据库
- **OLTP vs OLAP**：行存储 vs 列存储；压缩与向量化执行

### Part II: 分布式数据

### 4. 数据复制（Replication）
- **单主复制**：写到主库，复制到从库；主库故障需要故障转移
- **复制滞后问题**：
  - 读自己写（Read-after-write）一致性
  - 单调读（Monotonic reads）
  - 一致性前缀读
- **多主复制**：写入冲突处理（Last Write Wins / 合并函数 / 用户解决）
- **无主复制（Dynamo 风格）**：法定人数（Quorum）读写；Sloppy Quorum；反熵

### 5. 数据分区（Partitioning）
- **按键范围分区**：有序，支持范围查询；易出现热点
- **按键 Hash 分区**：均匀分布；失去范围查询能力
- **二级索引**：本地索引（写一次，读多个分区）vs 全局索引（写多次，读一次）
- **再平衡**：固定分区数 / 动态分裂 / 按节点比例

### 6. 事务（Transactions）
- **ACID 的真实含义**
  - Atomicity：要么全完成要么全不完成（与并发无关）
  - Consistency：应用程序不变量（不是数据库保证的）
  - Isolation：并发事务不互相影响
  - Durability：提交的数据不丢失
- **弱隔离级别**：
  - Read Uncommitted → Read Committed → Repeatable Read → Serializable
  - 脏读 / 不可重复读 / 幻读 / 写偏差（Write Skew）
- **序列化实现**：两阶段锁（2PL）/ 可串行化快照隔离（SSI）

### 7. 分布式系统的挑战
- **部分失败**是分布式系统的本质——无法区分"慢"和"挂了"
- **时钟不可靠**：NTP 误差；挂钟 vs 单调时钟；逻辑时钟（Lamport Clock）
- **网络不可靠**：消息可以延迟、丢失、重复；Timeout 是唯一探测手段

### 8. 一致性与共识
- **线性一致性（Linearizability）**：最强的一致性保证，系统看起来只有一个副本
- **因果一致性**：比线性弱，但开销低；版本向量跟踪因果关系
- **分布式事务与共识**：2PC 的问题（协调者单点故障）；Paxos / Raft
- **Epoch 和 Leader 选举**：ZooKeeper / etcd 的核心价值

### Part III: 派生数据

### 9. 批处理（Batch Processing）
- MapReduce 编程模型：分布式排序 + Group By
- 数据流水线：多个 MapReduce 作业链接
- Spark/Flink 的改进：内存计算、迭代算法支持

### 10. 流处理（Stream Processing）
- **事件流**：数据库写操作 = 事件流；CDC（Change Data Capture）
- **流处理语义**：至少一次 / 至多一次 / 恰好一次
- **窗口操作**：滚动窗口 / 滑动窗口 / 会话窗口
- **流表二元性**：流是表的差分，表是流的积分

### 11. 数据系统的未来
- Lambda 架构（批量层 + 速度层 + 服务层）的问题与 Kappa 架构的改进
- 数据库解构：功能解绑后按需组合

## 🔑 关键概念

### 线性一致性 vs 因果一致性
**定义**: 
- 线性一致性：所有操作看起来是原子的，有全局顺序（最强）
- 因果一致性：保留因果关系，允许并发操作有不同顺序（较弱但可用）

**重要性**: 线性一致性代价极高（需要协调所有副本），因果一致性在大多数场景已足够

**应用场景**:
1. 银行扣款、库存扣减 → 需要线性一致性
2. 社交评论、点赞 → 因果一致性足够
3. 购物车 → 可以用 CRDT（冲突免费数据类型）

---

### LSM Tree vs B-Tree
**定义**: 两种存储引擎的核心数据结构，决定读写性能特征

**重要性**: 选型直接影响工作负载的性能表现

**应用场景**:
1. 写密集型（日志、时序、IoT）→ LSM Tree（Cassandra/RocksDB）
2. 读密集型 OLTP → B-Tree（MySQL/PostgreSQL）
3. 混合负载 → 需要测试基准

**注意事项**: LSM 有写放大和读放大问题，B-Tree 有碎片化问题

## 💡 实战应用场景

### 场景 1: 选型多租户数据存储方案
**背景**: FDE 需要为客户的多租户 SaaS 系统选型数据库

**解决方案**: 按工作负载特征和一致性需求选型

**实施步骤**:
1. 明确读写比例和延迟要求
2. 确认一致性要求（账务类 vs 内容类）
3. 估算数据规模，判断是否需要分区
4. 根据上述条件对照选型决策树

**预期效果**: 有理有据的选型，能向客户 CTO 说明权衡

---

### 场景 2: 诊断数据库复制延迟问题
**背景**: 用户反馈"刚提交的数据立刻刷新后看不到"

**解决方案**: 识别复制滞后，实现"写后读一致性"

**实施步骤**:
1. 读自己写的数据时，路由到主库而非从库
2. 或者记录最后写入时间戳，从库追上后再读
3. 长期：考虑增加从库数量或升级主从同步为半同步

**预期效果**: 用户不再出现"幻消失"体验
