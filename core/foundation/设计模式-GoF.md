# 《设计模式：可复用面向对象软件的基础》- GoF

## 📖 基本信息
- **作者**: Erich Gamma, Richard Helm, Ralph Johnson, John Vlissides（"四人帮"Gang of Four）
- **出版年份**: 1994
- **核心主题**: 23 种可复用的面向对象设计模式，以及选择和使用它们的方法论
- **目标读者**: 有面向对象编程基础，希望提升设计能力的中高级开发者
- **阅读难度**: intermediate

## 🎯 核心价值
> 模式不是发明，而是发现——设计模式将软件设计中反复出现的解决方案结晶为共同语言，让团队沟通从描述代码上升到描述意图。

## 📚 知识框架

### 主要章节/模块

### 1. 创建型模式（Creational Patterns）— 控制对象创建

| 模式 | 意图 | 典型场景 |
|------|------|---------|
| **Abstract Factory** | 创建一族相关对象，不指定具体类 | UI 主题系统、数据库适配层 |
| **Builder** | 分步构造复杂对象 | SQL Query Builder、配置对象构造 |
| **Factory Method** | 延迟到子类决定实例化哪个类 | 框架中的创建钩子 |
| **Prototype** | 通过克隆创建对象 | 对象池、撤销/重做系统 |
| **Singleton** | 确保类只有一个实例 | 配置管理、日志记录器 |

### 2. 结构型模式（Structural Patterns）— 组合对象和类

| 模式 | 意图 | 典型场景 |
|------|------|---------|
| **Adapter** | 将不兼容接口转换为兼容接口 | 第三方库封装、遗留系统集成 |
| **Bridge** | 将抽象与实现分离，独立变化 | 跨平台 UI、多数据库支持 |
| **Composite** | 树形结构，统一叶子和容器 | 文件系统、UI 组件树、菜单 |
| **Decorator** | 动态添加职责，比继承更灵活 | Java I/O 流、HTTP 中间件 |
| **Facade** | 为复杂子系统提供简单接口 | SDK 封装、领域服务入口 |
| **Flyweight** | 共享细粒度对象，节省内存 | 字体渲染、游戏粒子系统 |
| **Proxy** | 为对象提供代理，控制访问 | 延迟加载、权限控制、缓存 |

### 3. 行为型模式（Behavioral Patterns）— 对象间通信与职责分配

| 模式 | 意图 | 典型场景 |
|------|------|---------|
| **Chain of Responsibility** | 请求沿链传递，直到被处理 | 审批流、中间件管道 |
| **Command** | 将请求封装为对象，支持撤销 | 任务队列、宏命令、事务 |
| **Interpreter** | 定义语言文法并解释句子 | 规则引擎、DSL 解析 |
| **Iterator** | 顺序访问集合，不暴露内部 | 所有集合遍历 |
| **Mediator** | 用中介对象封装对象交互 | 聊天室、MVC 控制器 |
| **Memento** | 不破坏封装的前提下保存/恢复状态 | 编辑器撤销、游戏存档 |
| **Observer** | 一对多依赖，状态变化时通知 | 事件系统、响应式编程、MQ |
| **State** | 对象随内部状态改变行为 | 订单状态机、TCP 连接状态 |
| **Strategy** | 封装可互换的算法族 | 排序策略、支付方式、压缩算法 |
| **Template Method** | 定义算法骨架，子类实现步骤 | 框架钩子、数据处理流水线 |
| **Visitor** | 在不修改类的前提下添加操作 | 编译器 AST 遍历、报表生成 |

## 🔑 关键概念

### 如何选择模式
**定义**: 按"意图"而非"结构"选择模式——先确定你要解决什么问题，再找匹配的模式

**重要性**: 滥用模式比不用更危险；模式是对话工具，不是炫技手段

**应用场景**:
1. 有条件分支做类型判断 → Strategy 或 State
2. 需要扩展但不想改已有代码 → Decorator 或 Visitor
3. 对象创建逻辑复杂 → Factory Method 或 Builder

**注意事项**: 不要为了用模式而用模式，"简单直接"永远是最优先的选项

---

### 组合优于继承
**定义**: 通过持有对象引用复用功能，而不是继承父类

**重要性**: 继承是编译时固定的，组合是运行时可变的；继承破坏封装，组合保持封装

**应用场景**:
1. 功能组合呈爆炸式（颜色×形状×大小）→ 改用组合
2. 子类需要改变父类行为细节 → 考虑 Strategy 替代继承
3. 框架扩展点设计 → 接口 + 组合

## 💡 实战应用场景

### 场景 1: 支付方式扩展（Strategy 模式）
**背景**: 系统支持多种支付方式，新增支付方式需要修改核心流程

**解决方案**: 将支付算法抽象为 `IPaymentStrategy` 接口，每种方式独立实现

**实施步骤**:
1. 定义 `IPaymentStrategy` 接口：`void pay(amount)`
2. 实现 `WechatPayStrategy`, `AlipayStrategy`, `StripeStrategy`
3. `OrderService` 持有策略引用，运行时注入具体策略

**预期效果**: 新增支付方式只需新增类，不改已有代码（OCP）

---

### 场景 2: 权限控制和缓存（Proxy 模式）
**背景**: 需要在不修改业务逻辑的前提下添加缓存和权限检查

**解决方案**: 用代理对象包装真实对象，AOP 思想的基础

**实施步骤**:
1. 定义共同接口 `IUserService`
2. 创建 `UserServiceProxy`，内部持有真实 `UserService`
3. 代理在调用真实方法前/后插入权限检查和缓存逻辑

**预期效果**: 业务代码零感知，横切关注点集中管理

---

### 场景 3: 多步骤对象构造（Builder 模式）
**背景**: 查询对象有十几个可选参数，构造函数参数列表混乱

**解决方案**: Builder 模式 + 链式调用

**实施步骤**:
1. 内部类 `QueryBuilder` 持有所有参数
2. 每个参数对应一个返回 `this` 的 setter
3. `build()` 方法验证参数并创建最终对象

**预期效果**: 调用侧语义清晰：`new QueryBuilder().table("users").where("active=1").limit(20).build()`

## 📝 个人笔记

### 重要洞察
- "针对接口编程，而不是针对实现编程"是贯穿全书的核心原则
- 23 个模式并非全部需要记忆，核心高频模式约 8 个：Strategy、Observer、Decorator、Factory Method、Singleton、Command、Composite、Proxy
- 模式往往是成对出现的：Factory + Strategy、Composite + Iterator、Decorator + Proxy
