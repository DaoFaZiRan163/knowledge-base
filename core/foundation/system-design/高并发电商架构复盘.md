# 500万用户电商平台的高并发架构复盘

## 📊 基本信息

| 指标 | 数值 |
|------|------|
| 平台用户规模 | 500 万 |
| 峰值 QPS | 800+ |
| 可用性目标 | 99.9% |

## 🎯 核心观点

电商平台并发压力集中在：
- 活动/上新
- 支付
- 库存变化
- 消息通知

> "架构目标不是永远不出问题，而是问题出现时能扛住、能降级、能定位、能恢复。"

## 🏗️ 五层保护架构

```
入口层（限流、鉴权、黑白名单、基础参数校验）
    ↓
缓存层（承接高频读请求，处理穿透、击穿、雪崩）
    ↓
服务层（拆分核心服务边界，避免单点拖累）
    ↓
异步层（通知、统计、日志、非实时状态同步）
    ↓
数据层（保护核心表，做好读写分离和归档）
```

## 💾 缓存设计原则

| 类型 | 处理方式 | 说明 |
|------|---------|------|
| **适合缓存** | 商品、配置 | 更新频率低，读多写少 |
| **谨慎缓存** | 用户状态 | 需明确一致性边界 |
| **不能只靠缓存** | 库存、支付 | 必须回到核心交易链路 |

**关键问题：**
- 缓存失效时数据库能否承受？
- 热点 key 如何处理？
- 数据错误如何快速修正？

## 🛡️ 数据库保护策略

| 问题 | 处理方式 | 收益 |
|------|---------|------|
| 高频读 | 缓存+读库+预聚合 | 降低主库压力 |
| 慢查询 | 索引治理+SQL审查+分页限制 | 减少连接占用 |
| 大表增长 | 冷热分离+归档策略 | 保持核心表轻量 |
| 写入尖峰 | 队列削峰+批量处理 | 保护核心事务 |

## ⚡ 异步化核心要点

**区分两类动作：**
1. **用户必须立即感知** → 同步处理
2. **可延迟** → 异步处理

**必须补上：**
- 幂等
- 重试
- 死信队列
- 补偿机制

## 🔻 限流与降级设计

**优先级（核心交易优先）：**
| 优先级 | 类型 | 处理 |
|--------|------|------|
| P0 | 核心交易 | 最高保障 |
| P1 | 查询 | 可降级 |
| P2 | 推荐 | 可关闭 |
| P3 | 统计 | 可延迟 |

> "降级不是失败，而是主动保住核心体验。"

## 📊 监控体系

**四类指标：**
1. **业务指标** —— 订单量、转化率
2. **接口指标** —— QPS、延迟、错误率
3. **资源指标** —— CPU、内存、连接数
4. **链路追踪** —— 请求路径、日志关联

> "只有数据齐全，故障复盘才不是猜测。"

## 📈 核心结论

> "高并发架构不是一次性设计完成的，是在真实流量、故障复盘和持续优化中迭代出来的。"

**真正可靠的系统，一定有：**
- 边界
- 保护
- 监控
- 演练

能承受峰值、能局部降级、能快速恢复，才是长期稳定的关键。
