# 为什么我选择 .NET 9 + CQRS 做 AI 微服务

## 🎯 核心洞察

> "AI 微服务的核心不是'调用大模型'，而是把不确定的智能能力放进可控的工程结构里。"

## 🤔 为什么选 .NET 9

| 考量 | 分析 |
|------|------|
| **团队能力匹配** | 团队长期使用 C# 和 .NET，有深厚积累 |
| **工程稳定性 > 语言流行度** | 企业内部系统，稳定性优先 |
| **性能足够** | AI 瓶颈在模型响应、检索、外部工具调用、网络延迟 |
| **类型系统清晰** | 复杂请求易于建模 |
| **生态完整** | 认证、配置、日志、监控、消息队列、数据库访问全覆盖 |

## 🏗️ 为什么加 CQRS

**CQRS 分离读写复杂度：**

AI 系统中有大量：
- **读操作**：知识库、会员信息、订单、对话历史、规则
- **写操作**：保存会话、记录评估、创建工单、写入反馈

**不加 CQRS 的三个问题：**
1. Query 侧越来越重
2. Command 侧副作用累积
3. 测试困难

### CQRS 组件

| 组件 | 职责 | 特点 |
|------|------|------|
| **Query** | 读知识库、会话、会员、订单、配置 | 无副作用、可缓存、可追踪 |
| **Command** | 保存会话、写反馈、创建工单、记录评估 | 每个动作有明确业务含义 |
| **Handler** | 收敛每个请求到独立 Handler | 减少大 Service 类，便于单元测试和灰度替换 |
| **Pipeline** | 统一处理日志、认证、参数校验、异常、时间统计、追踪 | 标准化横切关注点 |

## 🎯 AI 微服务边界设计

**应该包含的职责：**
- 智能问答
- 意图识别
- RAG 编排
- 工具选择
- 结果评估
- 对话记忆

**不应该包含的职责：**
- 订单状态
- 会员等级
- 售后流程

> "这些留在业务系统里，通过稳定 API 与 AI 服务协作。"

**收益：**
1. AI 服务不会复制业务系统数据规则，避免"两个真相来源"
2. 业务系统不会被 AI 实验节奏拖累

## 📁 项目结构

```
AiService.Api           # 外部接口
AiService.Application   # CQRS Handlers 和业务用例
AiService.Domain        # 领域对象和规则
AiService.Infrastructure# 数据库、向量库、模型提供商、外部业务 API
AiService.Workers       # 异步评估、对话摘要、样本积累
```

**核心原则：** "让每个模块知道自己负责什么，也知道自己不负责什么。"

## 🔍 可观测性优先

AI 服务最难排查的不是"API 报错"，而是"回答不正确"。完整链路必须能回放。

**关键可观测元素：**

| 元素 | 说明 |
|------|------|
| **TraceId** | 贯穿 AI 请求全步骤 |
| **Prompt 版本** | 使用的 prompt 模板和版本 |
| **检索证据** | 召回的文档、分数、来源、被截断的上下文 |
| **工具调用** | 工具名、参数、耗时、结果、失败原因 |
| **评估结果** | 用户反馈、自动评分、人工转交状态 |

## ⚖️ 技术选型权衡

| 方案 | 未选原因 |
|------|---------|
| **Python FastAPI** | AI 生态丰富、原型速度快，但团队主力不是 Python，长期维护和业务系统整合成本更高 |
| **Node.js** | 接口开发快、前后端协作轻量，但强类型和复杂业务建模不是团队最熟悉的路径 |
| **直接集成进主业务系统** | 短期整合最快，但 AI 迭代频率高，容易污染主系统边界，灰度和回滚成本高 |

## 💡 架构哲学

> "AI 服务不是越智能越好，而是越可解释、可回滚、可定位越好。"

在企业系统中，黑盒能力必须被工程日志和流程边界约束。

## 📊 核心结论

> "技术选型最终要回答的不是'流不流行'，而是'半年后线上出问题时，团队能不能快速定位并修好'。"
