# 《系统设计面试》- Alex Xu & Sahn Lam

## 📖 基本信息
- **作者**: Alex Xu（Volume 1）, Alex Xu & Sahn Lam（Volume 2）
- **出版年份**: 2020（Vol.1）, 2022（Vol.2）
- **核心主题**: 大规模分布式系统的设计方法论，覆盖从需求分析到组件选型的完整流程
- **目标读者**: 准备系统设计面试的工程师；需要为客户设计可扩展架构的 FDE
- **阅读难度**: intermediate

## 🎯 核心价值
> 系统设计没有标准答案——训练的是在约束条件下做权衡决策的结构化思维，而非背诵架构图。

### 主要章节/模块

### 1. 系统设计面试框架（4步法）
1. **理解问题，明确约束**（3-5 分钟）
   - 功能性需求：系统做什么
   - 非功能性需求：规模、延迟、可用性、一致性
   - 估算 QPS、存储量、带宽
2. **高层设计**（10-15 分钟）
   - 画出核心组件：Client → LB → App → DB/Cache
   - 说明数据流
3. **深入细节**（15-20 分钟）
   - 瓶颈在哪里？如何解决？
   - 数据库设计、API 设计、核心算法
4. **总结**（3-5 分钟）
   - 权衡取舍、可能的故障点、扩展方向

### 2. 核心组件与权衡

#### 扩展性基础
- **垂直扩展 vs 水平扩展**：垂直有上限；水平扩展需要无状态
- **负载均衡**：Round Robin / 最少连接 / IP Hash（会话粘连）
- **CDN**：静态资源分发，减轻源站压力；Push vs Pull CDN
- **无状态 Web 层**：Session 移到 Redis，实现水平扩展

#### 数据库扩展
- **Read Replica（主从复制）**：读写分离，适合读多写少
- **分片（Sharding）**：水平拆分，按 User ID / Hash / Range
- **分片键选择**：均匀分布、最小化跨分片查询
- **分片问题**：Celebrity 问题、Join 困难、重新分片

#### 缓存策略
| 策略 | 适用场景 | 风险 |
|------|---------|------|
| Cache-aside（旁路缓存）| 读多写少 | 冷启动 |
| Write-through | 写后立即更新缓存 | 写延迟增加 |
| Write-behind | 异步写入 DB | 数据可能丢失 |
| Refresh-ahead | 预测性预热 | 可能缓存无用数据 |

### 3. 经典系统设计案例

#### 短链接服务（TinyURL）
- 核心：长 URL → 短 ID 的映射
- ID 生成：Base62 编码（62^7 ≈ 3500亿），全局 ID 生成器
- 读多写少，缓存命中率高
- 301 vs 302 重定向（301 永久/节省流量 vs 302 临时/便于统计）

#### 新闻推送（News Feed）
- **推模式（Fan-out on write）**：写时将内容推给所有粉丝，读快但写慢
- **拉模式（Fan-in on read）**：读时聚合关注者内容，写快但读慢
- **混合模式**：普通用户推，大 V 拉

#### 通知系统
- 多渠道：Push / Email / SMS
- 队列解耦，每个渠道独立的 Worker Pool
- 消息去重（Message ID + 幂等消费）
- 优先级队列处理紧急通知

#### 限速器（Rate Limiter）
| 算法 | 特点 |
|------|------|
| Token Bucket | 允许突发，平均限速 |
| Leaky Bucket | 平滑输出，不允许突发 |
| Fixed Window | 简单，边界问题 |
| Sliding Window Log | 精确，内存消耗大 |
| Sliding Window Counter | 精确且内存高效 |

#### 分布式唯一 ID 生成器
- Twitter Snowflake：时间戳(41) + 机器ID(10) + 序列号(12)
- 优点：有序、高性能、去中心化
- 时钟回拨问题的处理

#### 搜索自动补全
- Trie 数据结构 + 缓存热词
- 每周离线重建 Trie，定时同步到缓存
- 前缀哈希表优化查询

#### 视频流（YouTube/Netflix）
- 视频上传：分片上传、断点续传、转码队列
- 视频播放：CDN 分发、自适应码率（ABR）
- 推荐系统：协同过滤、内容相似度

### 4. 关键技术选型决策树

```
数据存储选型：
├── 需要事务/强一致？ → 关系型数据库（MySQL/PostgreSQL）
├── 海量读写/简单查询？ → NoSQL（Cassandra/DynamoDB）
├── 键值查询为主？ → Redis / DynamoDB
├── 全文搜索？ → Elasticsearch
├── 时序数据？ → InfluxDB / TimescaleDB
└── 图关系查询？ → Neo4j

消息队列选型：
├── 需要持久化/回放？ → Kafka
├── 任务队列/简单场景？ → RabbitMQ / Redis Queue
└── 顺序消费？ → Kafka（单分区）
```

## 🔑 关键概念

### CAP 定理
**定义**: 分布式系统在网络分区（P）发生时，只能在一致性（C）和可用性（A）中选一个

**重要性**: 指导数据库和缓存系统的选型决策

**应用场景**:
1. 银行转账 → CP（强一致，可以短暂不可用）
2. 社交媒体 Feed → AP（高可用，最终一致）
3. 购物车 → AP + 最终一致

**注意事项**: 实际上是 PACELC 定理更准确：即便无网络分区，也需要在延迟和一致性间权衡

---

### 一致性哈希（Consistent Hashing）
**定义**: 哈希环上均匀分布节点，节点增减时只迁移相邻数据

**重要性**: 解决普通哈希在节点变化时大规模数据迁移问题

**应用场景**:
1. 分布式缓存（Memcached 集群）
2. 数据库分片路由
3. 负载均衡器

**注意事项**: 虚拟节点解决数据分布不均问题

## 💡 实战应用场景

### 场景 1: 客户要求系统支持百倍用户增长
**背景**: 现有单机系统需要扩展到支持千万级用户

**解决方案**: 分层扩展策略

**实施步骤**:
1. 加缓存层（Redis），命中率 80%+ 可减轻 DB 压力
2. 数据库读写分离，读 Replica 处理查询
3. 无状态 Web 层 + 负载均衡水平扩展
4. 数据量大时分库分表（按业务或 Hash）

**预期效果**: 每步可支撑 5-10 倍增长，按需逐步实施

---

### 场景 2: 系统设计汇报给客户 CTO
**背景**: FDE 需要在白板前向技术决策者说明架构方案

**解决方案**: 4 步框架 + 主动说出权衡

**实施步骤**:
1. 先对齐需求：QPS 多少？峰值几倍？一致性要求？
2. 画高层图，说明数据流向
3. 识别瓶颈并说明解决方案（不需要提所有细节）
4. 主动说出方案的取舍："这个方案牺牲了强一致性，换取高可用"

**预期效果**: 展示结构化思维，而非背答案
