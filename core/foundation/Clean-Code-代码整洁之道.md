# 《代码整洁之道》- Robert C. Martin

## 📖 基本信息
- **作者**: Robert C. Martin（Uncle Bob）
- **出版年份**: 2008
- **核心主题**: 如何编写可读、可维护、可扩展的高质量代码
- **目标读者**: 有一定经验的开发者，希望提升代码质量和专业素养
- **阅读难度**: intermediate

## 🎯 核心价值
> 代码是写给人读的，顺便让机器执行——整洁代码的本质是对未来读者（包括自己）的尊重。

## 📚 知识框架

### 主要章节/模块

### 1. 有意义的命名
- 名称应揭示意图：`int d` → `int elapsedTimeInDays`
- 避免误导性命名（别用 `accountList` 存非 List 结构）
- 使用可发音、可搜索的名称
- 类名用名词，方法名用动词
- 每个概念只用一个词（不要混用 `fetch/retrieve/get`）

### 2. 函数
- **短小**：函数应该只做一件事，20 行以内为宜
- **单一职责**：如果函数名需要加"And/Or"，说明做了多件事
- **参数数量**：0 个最好，1-2 个可接受，3 个需要警惕，3 个以上考虑封装成对象
- **无副作用**：函数要么查询（Query），要么命令（Command），不要混用
- **DRY 原则**：消除重复是代码整洁最重要的手段

### 3. 注释
- **好注释**：法律版权、意图说明、警告、TODO
- **坏注释**：重复代码内容的注释、误导性注释、注释掉的代码（直接删）
- 能用代码表达的就不要用注释——好的命名比注释更可靠

### 4. 格式
- **垂直格式**：相关代码放在一起，概念间用空行分隔
- **变量声明靠近使用位置**；实例变量在类顶部
- **相关函数就近**：调用者在被调用者上方
- **水平格式**：单行不超过 120 字符；用空格体现优先级

### 5. 对象和数据结构
- **数据结构**（DTO）：暴露数据，没有行为——适合过程式操作
- **对象**：隐藏数据，暴露行为——适合多态扩展
- **得墨忒耳律（Law of Demeter）**：只和直接朋友说话，避免 `a.getB().getC().doSomething()`
- **避免混合体**：半是对象半是数据结构的设计最糟糕

### 6. 错误处理
- **使用异常而非错误码**：错误码要求调用者立即检查
- **先写 try-catch-finally**：帮助定义作用域和契约
- **使用 Unchecked Exception**：Checked Exception 破坏封装
- **别返回 null，别传 null**：用特殊对象（Null Object 模式）或抛异常代替
- **异常要提供上下文信息**：失败的意图 + 失败的类型

### 7. 边界（第三方代码集成）
- 用**适配器/包装器**隔离第三方代码，避免第三方 API 渗透到业务逻辑
- 先为第三方库写**学习测试**（Learning Tests），理解其行为
- 对未完成的部分定义**接口**，用 Fake 实现推进开发

### 8. 单元测试（TDD 三定律）
1. 不允许编写任何生产代码，除非是为了通过一个失败的单元测试
2. 不允许编写超过一个失败单元测试的代码
3. 不允许编写超过足以通过当前失败测试的生产代码

- **FIRST 原则**：Fast / Independent / Repeatable / Self-Validating / Timely
- **每个测试一个概念**，测试代码和生产代码同等重要
- **整洁测试 = 可读性**，Build-Operate-Check 模式

### 9. 类
- **单一职责原则（SRP）**：类只有一个修改原因
- **内聚性**：类的方法操作的实例变量越多越内聚
- **开闭原则（OCP）**：对扩展开放，对修改封闭
- 当类变大时，先问"为什么会有多个修改原因"，而不是"怎么拆分代码"

### 10. 系统
- **构建与使用分离**：main 负责构建，业务代码不 new 依赖
- **依赖注入（DI）**：把依赖关系的控制权移交给调用者
- **AOP 思想**：横切关注点（日志、事务、安全）应该独立于业务逻辑
- 系统也应该从简单开始，演化而非预先设计

### 11. 迭进（简单设计四规则，Kent Beck）
1. 通过所有测试
2. 没有重复代码
3. 体现程序员意图
4. 最小化类和方法数量

### 12. 并发编程
- 并发解耦了**做什么**和**何时做**，但引入了复杂性
- **单一职责**：并发代码和其他代码分离
- **限制数据共享**：减少共享数据的使用
- **使用线程安全集合**，了解 `synchronized`/`volatile` 的含义
- 并发 Bug 难以重现，需要专门的测试策略

### 13. 逐步改进（案例：Args 解析器）
- 先让代码能工作，再让代码整洁——但不要拖太久
- 混乱堆积的代价远大于边写边整理

### 14. 代码的气味（Code Smells）与启发
**注释类**：过时注释、注释掉的代码、信息量不足的注释  
**环境类**：构建需要多步、测试需要多步  
**函数类**：过多参数、输出参数、标志参数、死函数  
**一般性问题**：一个源文件多种语言、明显的行为未实现、重复代码、抽象层次不对  
**名称类**：名称描述不清、名称在不同语境含义不同

## 🔑 关键概念

### 童子军规则
**定义**: 让营地比你来时更干净——每次提交代码都比签出时整洁一点点

**重要性**: 防止代码腐烂的最实际手段，不需要大规模重构

**应用场景**:
1. 修 Bug 时顺手改掉函数名
2. 加功能时顺手提取重复逻辑
3. Code Review 时发现坏命名就重命名

**注意事项**: 不要以"整洁"为由引入大范围变更影响测试

---

### 单一职责原则（SRP）
**定义**: 一个类（或函数）只有一个引起它变化的原因

**重要性**: 违反 SRP 导致"散弹枪手术"——一个需求改动要触碰多个文件

**应用场景**:
1. 一个类同时处理 DB 读写 + 业务逻辑 → 拆分
2. 一个函数既格式化数据又保存数据 → 拆分
3. Controller 里写了 SQL → 拆分出 Repository

**注意事项**: SRP 的"职责"指的是变化原因，不是功能多少

---

### 得墨忒耳律（Law of Demeter）
**定义**: 一个模块不应了解它所操作对象的内部细节，只和直接朋友通信

**重要性**: 防止链式调用造成的紧耦合，降低修改波及范围

**应用场景**:
1. 避免 `order.getCustomer().getAddress().getCity()`
2. 改为 `order.getCustomerCity()` 或让 Order 封装该逻辑
3. Data Transfer Object (DTO) 是例外，可以直接访问属性

**注意事项**: 不是禁止所有链式调用，Builder 模式和 Stream 操作是合理的流式调用

---

### 开闭原则（OCP）
**定义**: 软件实体应该对扩展开放，对修改封闭

**重要性**: 新增功能不需要修改已有代码，避免引入新 Bug

**应用场景**:
1. 用多态替代 if-else/switch 做类型分发
2. 用策略模式替代条件分支
3. 插件化架构设计

**注意事项**: OCP 的代价是更多的抽象层，只在真正会变化的点上使用

## 💡 实战应用场景

### 场景 1: Code Review 发现命名问题
**背景**: 接手遗留代码，函数名和变量名晦涩难懂

**解决方案**: 按"揭示意图"原则系统性重命名

**实施步骤**:
1. 用 IDE 的 Rename 重构，不要手动查找替换
2. 变量名改为说明"是什么"而非"怎么存"
3. 函数名改为说明"做什么"，布尔函数用 `isXxx/hasXxx/canXxx`

**预期效果**: 读代码不再需要注释辅助理解

---

### 场景 2: 函数越来越长
**背景**: 一个函数随需求增加膨胀到 100+ 行

**解决方案**: 提取函数（Extract Method）+ 单一职责拆分

**实施步骤**:
1. 识别函数内"段落"——自然分组的代码块
2. 每个段落提取为独立函数，用注释内容作为函数名
3. 检查是否有数据传递可以用对象封装

**预期效果**: 顶层函数只剩下调用序列，像一段高层语言描述的流程

---

### 场景 3: 错误处理逻辑污染业务代码
**背景**: 业务逻辑被大量 null check 和错误码判断淹没

**解决方案**: 异常替代错误码 + Null Object 模式

**实施步骤**:
1. 将返回错误码的函数改为抛出语义清晰的异常
2. 对"找不到"的场景用 Null Object 替代返回 null
3. 在系统边界集中处理异常，业务层保持干净

**预期效果**: 业务代码只描述"正常路径"，异常路径在统一位置处理

---

### 场景 4: 遗留代码集成第三方 SDK
**背景**: 引入新的第三方库，但不确定它的行为，担心后续难以替换

**解决方案**: 适配器模式隔离 + 学习测试先行

**实施步骤**:
1. 先写学习测试，覆盖你需要的功能点，理解 SDK 行为
2. 定义业务需要的接口（不是 SDK 的接口）
3. 用适配器实现该接口，内部调用 SDK

**预期效果**: 替换 SDK 只需改适配器，业务代码零改动

## 📝 个人笔记

### 重要洞察
- 整洁代码不是风格问题，是**可维护性和可读性**的工程问题，有可量化的成本
- "让代码能工作"和"让代码整洁"是两个独立的技能，都需要刻意练习
- 测试是重构的安全网——没有测试就没有勇气整理代码
- 命名是最廉价但回报最高的整洁代码投资
