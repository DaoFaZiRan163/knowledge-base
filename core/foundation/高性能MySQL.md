# 《高性能 MySQL》- Baron Schwartz et al.

## 📖 基本信息
- **作者**: Baron Schwartz, Peter Zaitsev, Vadim Tkachenko
- **出版年份**: 第三版 2012，第四版 2021
- **核心主题**: MySQL 性能优化的系统方法——从架构设计到查询调优的完整指南
- **目标读者**: 使用 MySQL 的开发者、DBA、系统架构师
- **阅读难度**: intermediate

## 🎯 核心价值
> 性能优化不是猜测，是测量——先用工具找到真正的瓶颈，再有针对性地优化，而不是凭直觉改配置。

## 📚 知识框架

### 主要章节/模块

### 1. MySQL 架构与历史
- **连接管理**：每个连接一个线程，连接池减少开销
- **查询缓存**（已废弃）：缓存整个查询结果，表写入就失效
- **解析器 → 预处理器 → 查询优化器 → 执行引擎 → 存储引擎**
- **InnoDB vs MyISAM**：InnoDB 支持事务/行锁/MVCC，是现代选择

### 2. 性能基准测试
- **吞吐量**：单位时间完成的事务数（TPS）
- **响应时间**：用百分位数（p95, p99）而非平均值
- **并发性**：真正并发执行的线程数（不等于连接数）
- **工具**：`sysbench`（基准），`mysqlslap`（并发），`pt-query-digest`（慢查询分析）

### 3. Schema 设计优化

#### 数据类型选择原则
1. **尽量小**：用能存下数据的最小类型（TINYINT vs INT vs BIGINT）
2. **尽量简单**：整型优于字符串；IP 用 INT 存储
3. **避免 NULL**：NULL 使索引和查询复杂化，用 0 或空字符串代替
4. **VARCHAR vs CHAR**：变长 VARCHAR 节省空间，但更新可能造成碎片；CHAR 适合固定长度

#### 范式 vs 反范式
- 范式化：减少冗余，更新方便，查询需要 Join
- 反范式化：冗余数据避免 Join，读性能好；更新需维护多处

### 4. 索引设计（核心章节）

#### B+Tree 索引
- 数据按索引列有序存储，支持：等值查询、范围查询、排序、覆盖索引
- **聚集索引（Clustered Index）**：InnoDB 主键即数据文件，叶子节点存完整行
- **二级索引**：叶子节点存主键值，查询时需"回表"

#### 索引设计原则
1. **最左前缀匹配**：复合索引 `(a,b,c)`，查询必须从 `a` 开始才能用索引
2. **覆盖索引**：索引包含查询所需的所有列，避免回表（`EXPLAIN` 显示 `Using index`）
3. **索引选择性**：选择性高的列放前面（`COUNT(DISTINCT col)/COUNT(*)`）
4. **避免在索引列上运算**：`WHERE YEAR(created_at)=2024` 无法用索引

#### 常见索引失效场景
- 列参与计算或函数调用
- 使用 `!=`, `NOT IN`, `LIKE '%prefix'`（前缀通配）
- 隐式类型转换（字符串列用整数查询）
- OR 连接的条件未全部覆盖索引

### 5. 查询性能优化

#### EXPLAIN 解读
| 字段 | 关注点 |
|------|--------|
| type | ALL（全表）→ index → range → ref → eq_ref → const（由差到好）|
| key | 实际使用的索引 |
| rows | 估计扫描行数（越小越好）|
| Extra | Using filesort / Using temporary 是警告信号 |

#### 慢查询优化流程
1. 开启慢查询日志：`slow_query_log=1, long_query_time=1`
2. 用 `pt-query-digest` 分析，找出最耗时的查询
3. `EXPLAIN` 分析执行计划
4. 针对性加索引或改写查询

#### 常见查询优化手段
- **分页优化**：`LIMIT 100000,20` 会扫描 100020 行；改用"书签续读" `WHERE id > last_id LIMIT 20`
- **COUNT 优化**：`COUNT(*)` 统计行数，`COUNT(col)` 统计非 NULL 行数
- **JOIN 优化**：小表驱动大表；JOIN 列加索引
- **子查询 vs JOIN**：通常 JOIN 性能更好，子查询可能产生临时表

### 6. MySQL 高级特性

#### 事务与锁
- **MVCC（多版本并发控制）**：读不加锁，通过 undo log 读历史版本
- **锁类型**：行锁（InnoDB）/ 间隙锁（防止幻读）/ 意向锁
- **死锁**：两个事务互相等待对方的锁；InnoDB 自动检测并回滚代价小的事务

#### 主从复制
- **Binlog 格式**：Statement（SQL语句）/ Row（行变更，更可靠）/ Mixed
- **半同步复制**：主库等至少一个从库确认后再提交，减少数据丢失风险
- **GTID**：全局事务 ID，简化故障切换时的位点确认

### 7. 服务器配置与优化

#### 关键配置项
| 配置 | 说明 |
|------|------|
| `innodb_buffer_pool_size` | 最重要配置，通常设为可用内存的 70-80% |
| `innodb_log_file_size` | 影响崩溃恢复速度和写入性能 |
| `max_connections` | 过大会导致内存耗尽；配合连接池使用 |
| `innodb_flush_log_at_trx_commit` | 1=安全（每次提交刷盘）；2=性能（每秒刷盘）|

## 🔑 关键概念

### 覆盖索引（Covering Index）
**定义**: 查询所需的所有列都在索引中，无需回表查主键

**重要性**: 消除回表是单个查询优化中效果最显著的手段之一

**应用场景**:
1. 高频列表查询：`SELECT id, name, status FROM orders WHERE user_id=?`
2. 分页查询的 ID 列表先走覆盖索引
3. 统计查询：`SELECT COUNT(*), status FROM orders GROUP BY status`

**注意事项**: 索引越宽，写入开销越大；只覆盖高频查询的列

---

### MVCC（多版本并发控制）
**定义**: 读操作不加锁，通过读取历史快照实现隔离；写操作只与写操作冲突

**重要性**: InnoDB 高并发读写性能的基础；理解它才能理解隔离级别的实现

**应用场景**:
1. 可重复读（RR）：事务开始时的快照一致
2. 读已提交（RC）：每次读取最新快照

**注意事项**: 长事务会积累大量 undo log，导致 undo 表空间膨胀

## 💡 实战应用场景

### 场景 1: 慢查询诊断
**背景**: 生产环境某接口 P99 延迟 5 秒，用户反馈

**解决方案**: 系统化慢查询分析流程

**实施步骤**:
1. `pt-query-digest` 分析慢查询日志，定位 Top 5 慢查询
2. `EXPLAIN` 查看执行计划，确认是否走索引
3. 针对扫描行数多的查询，设计覆盖索引
4. 验证：重建索引后对比 `rows` 列

**预期效果**: 通常慢查询优化后延迟降低 10-100 倍

---

### 场景 2: 大表分页性能问题
**背景**: 用户列表第 100 页加载时间 3 秒，前 3 页几乎瞬间

**解决方案**: 书签分页替代 OFFSET 分页

**实施步骤**:
1. 返回结果时，同时返回最后一条记录的 ID
2. 下一页请求携带此 ID：`WHERE id > :last_id ORDER BY id LIMIT 20`
3. 复合排序时，需要将排序列 + 主键作为联合书签

**预期效果**: 分页查询时间恒定，不随页码增大而增加
