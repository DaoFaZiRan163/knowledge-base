## 今日学习内容

> MVP（Minimum Viable Product，最小可行产品）是精益创业方法论的核心工具，其本质不是"做一个功能简陋的产品"，而是"用最小成本验证最核心的商业假设"。对AI产品而言，MVP有其特殊性：AI能力本身可以作为验证假设的手段（如先用人工+AI工具模拟自动化功能），但AI回答质量的不稳定性也会干扰假设验证的结论。理解MVP的底层逻辑，是避免过度开发和正确解读验证结果的关键。

## 核心知识点

### 知识点一：MVP的本质——假设验证器

MVP最容易被误解的地方是"Minimum"这个词。正确理解：

| 误解 | 正确理解 |
|------|---------|
| 功能最少的产品 | 能验证核心假设的最精简方案 |
| 技术不完善的产品 | 刻意简化非核心部分，确保核心体验足够真实 |
| 一次性凑合 | 构建快速学习循环的起点 |
| 只给早期用户看的 | 面向真实目标用户的真实验证 |

**核心公式**：
```
MVP = 核心假设 + 最快的验证方式 + 可测量的成功标准
```

在构建MVP之前，必须先回答：**我们这次要验证的最重要的假设是什么？**

### 知识点二：MVP的类型谱系

MVP不一定是可交互的产品，可以有多种形态：

```
低成本  ←─────────────────────────→  高成本
低保真                                 高保真

问卷调研 → 落地页测试 → 人工模拟 → 原型测试 → 功能受限版本 → 完整MVP
```

**各类型适用场景**：
- **落地页测试**：验证用户对概念的兴趣（收邮箱=表达意向）
- **人工模拟（Wizard of Oz）**：后台人工完成AI的工作，验证用户是否接受这种交互方式
- **单功能版本**：只做最核心功能，验证核心价值假设
- **邀请制内测**：控制用户范围，获取高质量反馈

**AI产品特别推荐：Wizard of Oz MVP**
用人工+现有AI工具（如ChatGPT）模拟尚未开发的AI功能，既能快速验证用户需求，又避免过早投入算法开发成本。

### 知识点三：Build-Measure-Learn 循环

精益创业的核心方法论是快速迭代的"建造-测量-学习"循环：

```
         ┌─────────────────────┐
         │                     │
         ▼                     │
    [ BUILD ]              学到什么？
  创建MVP/实验          ─────────────→  [ LEARN ]
         │                             分析结果
         │                             更新假设
         ▼                                 │
    [ MEASURE ]  ←────────────────────────┘
   收集真实数据
   (行为数据>问卷)
```

**关键原则**：
1. 数据优先于直觉：用行为数据（用户真实点击、转化）而非调研问卷（用户说会做）
2. 迭代速度是竞争力：缩短每次循环的时间
3. 及时Pivot：当核心假设被证伪时，勇于调整方向

### 知识点四：AI产品MVP的特殊挑战

AI产品在做MVP验证时面临三个独特挑战：

**挑战1：AI质量不稳定干扰验证**
- 问题：用户可能因为AI回答质量差而流失，但你无法判断是"需求不存在"还是"AI做得不够好"
- 对策：在MVP阶段设置人工审核层，确保验证的是需求而非AI质量

**挑战2：冷启动的数据稀疏**
- 问题：AI产品效果往往依赖数据积累，MVP阶段数据不足导致效果差，影响假设验证
- 对策：先验证"用户愿意使用这种方式解决问题"，再验证"AI做得足够好"，分步验证

**挑战3："新奇效应"掩盖真实留存**
- 问题：用户因为AI新奇感而高频使用MVP，但新鲜感过后流失
- 对策：延长MVP观察周期（至少4周），重点看D30留存而非D1次日留存

## 关键概念详解

> 每个概念按 **是什么 → 为什么重要 → 具体例子** 展开。

### 核心假设（Leap-of-faith Assumption）
**是什么**：产品成功依赖的、尚未被验证的关键信念，通常关于用户需求或商业模式。
**为什么重要**：所有产品决策都建立在一系列假设之上，最危险的假设是那些被当作事实对待的假设。MVP的价值在于把假设显性化并快速验证。
**具体例子**：AI问答产品的核心假设——"企业员工愿意用AI替代向同事提问的习惯"。这个假设如果为真，产品就有市场；如果为假，再好的技术也没用。MVP应该先验证这个行为改变假设，而不是先打磨AI回答质量。

### Pivot（战略转向）
**是什么**：当核心假设被验证为错误时，保留已有学习成果，调整产品方向或商业模式的重大决策。
**为什么重要**：Pivot不是失败，是精益创业中正常的学习过程。没有数据支持的坚持才是真正的失败。AI赛道快速变化，能快速Pivot的团队往往比技术更强但执行更固执的团队走得更远。
**具体例子**：AI问答产品原本假设用户想要"一问一答"，MVP验证后发现用户更需要"主动推送相关知识"，于是从被动问答Pivot到主动推送知识流——这是一个成功的Pivot案例。

### 成功标准（Success Metrics）
**是什么**：在MVP实验开始前预先定义的、判断假设是否成立的量化指标。
**为什么重要**：没有预先定义成功标准，实验结束后很容易陷入"事后合理化"——对于任何数据都能找到一个解读角度说成功了。
**具体例子**：AI问答MVP的成功标准——"连续使用3周以上的用户占比 > 30%"或"用户每周自发使用次数 > 5次"。注意：成功标准必须在实验前定义，而不是看到数据后再设定门槛。

### 过早优化（Premature Optimization）
**是什么**：在核心价值假设还未验证的情况下，花费大量时间完善非核心功能或技术细节。
**为什么重要**：过早优化是创业公司最常见的死亡原因之一——团队花6个月把产品做得很完美，却发现市场根本不需要这个产品。
**具体例子**：AI问答产品在还不知道用户是否需要"记忆历史对话"的时候，就花了两个月开发复杂的记忆管理系统——这就是过早优化。正确做法是先用手动记录的方式验证用户是否在意记忆功能。

**复盘框架**：

```markdown
## AI问答产品MVP复盘

### 1. 当时的核心假设是什么？
（你们当时做MVP时，在验证哪个最关键的假设？）

### 2. 选择的MVP形态是什么？
（功能受限版本 / 人工模拟 / 单一场景试点 / 其他）

### 3. 验证结果如何？
- 成功标准预设（当时有没有预先定义？）
- 实际数据结果（关键指标是多少？）
- 假设是否被验证？

### 4. 基于MVP做了哪些决策？
（是否有Pivot？主要功能方向调整了什么？）

### 5. 回头看，MVP阶段最大的遗憾是什么？
（做了哪些过早优化？跳过了哪些关键验证？）

### 6. 如果重来，MVP策略会怎么改？
```

**参考示例答案片段**：
> 我们当时的核心假设是"员工在工作中遇到重复性问题时，愿意通过AI而非询问同事来获取答案"。MVP选择了单一场景（IT支持问题）的受限版本，成功标准是"30天内用户主动发起对话 > 3次"。结果验证了需求存在，但发现用户更在意回答速度而非准确性（与假设不同）。最大遗憾是过早投入了知识库管理UI的开发，这个功能在MVP阶段用户根本没用到。

## 面试话术

**Q：请介绍一下你们产品的MVP策略，以及你从中学到了什么。**
> 我们的MVP策略核心是"先验证需求假设，再投入技术建设"。我们选择了人工模拟AI的方式（Wizard of Oz），用现有工具模拟AI问答功能，重点验证用户是否愿意改变"向同事提问"的习惯。在预先定义的成功标准（30天内重复使用率>40%）达成后，才开始正式的AI能力开发。这个经历让我学到：AI产品最危险的假设不是技术能不能做到，而是用户愿不愿意改变行为习惯。

**Q：如果产品资源有限，你会如何决定MVP的范围？**
> 我会先列出产品成立的所有核心假设，按"验证的紧迫性"和"假设为假时的损失"排序，优先验证最核心、风险最高的假设。MVP的范围只包含验证该假设所必须的功能。其他功能无论多重要，在核心假设验证完成前都可以暂时搁置。在AI产品中，我特别会区分"AI需要做到多好"和"用户是否接受AI辅助"这两个假设——前者可以后验，后者必须先验。
