## 今日学习内容

> 用户故事（User Story）是敏捷开发中传递需求的基本单元。它不描述系统功能，而是描述"谁"在"什么情境下"想要"什么价值"。好的用户故事让研发直接感受到真实用户的目标，而不是机械地实现功能规格。对AI产品而言，用户故事还需要描述当AI表现不佳时用户的处理预期，这是传统用户故事不曾触及的新维度。

---

## 核心知识点

### 知识点一：用户故事的标准格式

用户故事的黄金句式：

```
As a [用户角色]
I want to [完成某个动作/达成某个目标]
So that [获得某种价值/解决某个问题]
```

**中文版**：
```
作为一个 [用户角色]，
我希望能够 [完成某个动作]，
这样我就可以 [获得某种价值]。
```

**AI产品示例**：
```
作为一个每天使用AI问答解决工作问题的产品经理，
我希望能够让AI记住我的职业背景和偏好，
这样每次提问时不需要重复解释自己的上下文，节省时间。
```

注意：用户故事不说"系统应该做xxx"，而是说"用户想要xxx"，两者视角完全不同。

### 知识点二：INVEST原则——好用户故事的六条标准

| 字母 | 原则 | 含义 | AI产品应用 |
|------|------|------|-----------|
| I | Independent | 独立的，尽量不依赖其他Story | 记忆查看与记忆删除应拆分为独立Story |
| N | Negotiable | 可协商的，不是固定的合同 | 记忆提取的触发时机可与研发协商 |
| V | Valuable | 对用户有价值 | 每条Story都要说清楚"So that"的价值 |
| E | Estimable | 可估算的，研发能估出工作量 | AI功能需配合技术可行性讨论 |
| S | Small | 小的，2-5天内可完成 | 把"完整的记忆系统"拆成多个小Story |
| T | Testable | 可测试的，有明确验收标准 | 每个Story需附带Given-When-Then的AC |

### 知识点三：验收标准（AC）的三种写法

**方法一：Given-When-Then（最推荐）**
```
Given：用户处于AI对话界面，已发送一条包含个人信息的消息
When：对话结束后，AI提取到了记忆条目
Then：用户在"我的记忆"页面可以看到这条新增的记忆条目
      且记忆条目显示提取时间和来源对话
```

**方法二：规则清单（适合状态类需求）**
```
- 记忆列表按创建时间倒序排列
- 每条记忆显示内容摘要（最多50字）和创建时间
- 记忆条数超过50条时，删除按钮置灰并提示"已达上限"
- 删除操作需二次确认弹窗
```

**方法三：场景矩阵（适合有多种分支的需求）**

| 场景 | 用户行为 | 系统响应 |
|------|---------|---------|
| 正常情况 | 点击删除按钮 | 弹出确认弹窗 |
| 确认删除 | 点击"确认删除" | 记忆条目从列表消失 |
| 取消删除 | 点击"取消" | 弹窗关闭，记忆保留 |
| 网络异常 | 点击删除，请求失败 | 提示"删除失败，请重试" |

### 知识点四：Epic、Story、Task的层级关系

```
Epic（史诗）：记忆管理功能
  └── Story 1：作为用户，我希望能查看AI记住了哪些关于我的信息
  └── Story 2：作为用户，我希望能编辑AI记忆中不准确的条目
  └── Story 3：作为用户，我希望能删除不想被记住的信息
  └── Story 4：作为用户，我希望AI在回答时引用我的记忆提供个性化回答
        └── Task 1：设计记忆列表UI
        └── Task 2：实现记忆提取接口
        └── Task 3：实现SystemPrompt记忆注入逻辑
```

- **Epic**：大功能模块，通常需要数周完成
- **Story**：用户可感知的最小价值单元，通常2-5天完成
- **Task**：技术任务，通常由研发自己拆分，1天内完成

### 知识点五：AI产品用户故事的特殊考量

AI产品的用户故事需要覆盖AI行为异常的场景，这是传统产品没有的：

**AI正常场景Story**：
```
作为用户，当我向AI提问时，我希望在3秒内看到AI开始流式输出，
这样我知道系统在响应，不会因等待而离开。
```

**AI异常场景Story**（容易被遗漏）：
```
作为用户，当AI回答与我的问题完全不相关时，
我希望有一个"重新回答"按钮让AI重试，
这样我不需要重新输入同样的问题。
```

```
作为用户，当AI给出的回答包含明显错误信息时，
我希望能够标记"这个回答有问题"，
这样我的反馈能帮助改善AI的回答质量。
```

### 知识点六：故事地图（Story Map）——让用户故事有结构

故事地图是把散乱的User Story组织成有结构视图的工具：

```
用户旅程主轴（横向）：
注册 → 首次对话 → 深度使用 → 分享/推荐

每个阶段的核心活动（纵向）：
[首次对话]
  必须有（MVP层）：发送问题 / 看到AI回答 / 重新提问
  应该有（下一版）：流式输出 / 复制回答 / 评价回答
  可以有（未来版）：语音输入 / 多轮对话引导 / 话题推荐
```

故事地图帮助PM看清楚哪些Story是MVP核心，哪些是增量迭代。

## 关键概念详解

### 用户故事（User Story）
**是什么**：以用户视角写成的需求描述，格式为"作为XX，我希望XX，这样我可以XX"。
**为什么重要**：用户故事将需求焦点从"系统功能"转移到"用户价值"，研发在实现时会始终思考"这对用户有什么意义"，减少为技术而技术的功能设计。
**具体例子**：传统需求写法："系统需要支持对话历史搜索功能"；用户故事写法："作为每天使用AI问答的职场人，我希望能搜索3个月内的对话记录，这样我能快速找回之前讨论过的方案而无需重新提问。"

### 验收标准（Acceptance Criteria, AC）
**是什么**：定义一个用户故事"完成"所需要满足的具体、可测量的条件列表。
**为什么重要**：AC是研发的终点线和测试的检查清单。没有AC，PM和研发对"完成"的定义不同，导致大量返工。
**具体例子**：对"用户可以删除记忆"这个Story，AC可以是："删除前弹出确认对话框 → 确认后记忆从列表消失且不可恢复 → 删除成功后展示Toast提示 → 删除失败时显示错误信息并保留原记忆。"

### 故事点（Story Point）
**是什么**：衡量用户故事"实现难度"的相对估算单位，通常用斐波那契数列（1、2、3、5、8、13）表示。
**为什么重要**：故事点不是工时，而是复杂度。它帮助团队在Sprint规划时评估一个迭代能完成多少工作量，逐渐建立团队的"速度"基准。
**具体例子**：AI问答"查看记忆列表"可能估算3点（相对简单），"AI自动提取记忆并去重"可能估算8点（涉及模型调用和逻辑判断），"记忆个性化回答效果评估"可能估算13点（需要A/B测试基础设施）。

### 最小可接受故事（Minimum Viable Story）
**是什么**：用户故事拆分的最小粒度原则——每个Story必须独立可交付且对用户有价值。
**为什么重要**：过大的Story无法在一个Sprint内完成，过小的Story则没有独立价值。找到这个平衡点是PM的核心能力之一。
**具体例子**："记忆管理"整个功能不能是一个Story，需要拆成"查看记忆"、"编辑记忆"、"删除记忆"、"AI对话中使用记忆"等多个独立Story，每个都可以单独上线并对用户有价值。

## 练习任务输出

**题目**：为AI问答产品写5条 User Story，包含完整的验收标准

**参考示例**（完整格式）：

```markdown
### Story 1：AI流式回答
作为频繁使用AI问答的用户，
我希望在发送问题后立即看到AI开始逐字输出回答，
这样我感知到系统在响应，不会因等待而感到焦虑。

**验收标准**：
- Given：用户发送一条问题
  When：后端收到请求并开始调用模型
  Then：前端在1.5秒内开始显示流式输出，光标闪烁
- Given：AI回答过程中
  When：用户点击"停止生成"按钮
  Then：生成立即停止，已输出内容保留，停止按钮消失
- Given：模型调用失败
  When：无法返回任何内容
  Then：3秒后显示"回答失败，请重试"，提供"重试"按钮

---

### Story 2：【请填写你的第2条Story】

### Story 3：【请填写你的第3条Story】

### Story 4：【请填写你的第4条Story】

### Story 5：【请填写你的第5条Story】
```

## 面试话术

**Q：你如何写用户故事？能举个AI产品的例子吗？**
> 我用标准的"As a / I want / So that"格式，确保每条Story以用户价值为核心而不是技术功能。以AI问答为例，我会写："作为高频用户，我希望AI能记住我的职业背景，这样不需要每次重复自我介绍"，然后为这个Story写Given-When-Then格式的验收标准，覆盖正常路径和AI失效的降级场景。

**Q：验收标准应该由谁来写？PM还是测试？**
> 这是共同责任——PM定义"什么算完成"，测试工程师补充"边界case和异常路径"，研发确认"技术可行性"。实践中，PM写初稿，Sprint规划会上三方一起审阅补充，形成最终版。这样做的好处是三方对"完成"的定义一致，大幅减少验收阶段的争议。

**Q：用户故事拆多大比较合适？**
> 原则是"一个Sprint内可以完成，且单独上线时对用户有价值"。实践中我用INVEST原则检查：特别关注"Small"（5天内完成）和"Valuable"（独立有价值）这两条。如果一个Story在Sprint内无法完成，需要继续拆分；如果拆完之后某个Story单独上线没有价值，需要合并或重组。
