## 今日学习内容

> PRD（Product Requirements Document，产品需求文档）是PM与研发、设计、测试团队沟通的核心载体。一份高质量的PRD不仅说清楚"做什么"，更要明确"为什么做"和"验收标准"。对AI产品而言，PRD还需要特别描述模型行为预期、Prompt策略与降级方案，这是传统PRD中从未涉及的新章节。写好PRD，是从"想法"到"落地"的关键一跳。

---

## 核心知识点

### 知识点一：PRD的本质是对齐契约

PRD不是技术文档，也不是产品故事，而是**多角色之间的对齐契约**。它的核心价值在于：

| 受众 | 需要从PRD获取的信息 |
|------|-------------------|
| 研发工程师 | 技术边界、接口定义、异常处理 |
| 设计师 | 用户流程、交互逻辑、视觉规范 |
| 测试工程师 | 验收标准、边界 case、性能指标 |
| 业务/运营 | 上线时间、功能范围、风险说明 |
| 产品自己 | 决策记录、优先级依据 |

一份PRD的质量，用一个问题就能检验：**研发不来问你，能否把功能做出来？** 如果答案是"是"，PRD就合格了。

写PRD时要克服两个常见错误：① 用UI图代替文字逻辑（UI会变，逻辑不变）；② 把"需求来源"写成"用户希望"而不是"业务目标"。

### 知识点二：PRD的标准结构

一份完整的PRD通常包含以下章节，顺序不可随意调换（因为后者依赖前者的上下文）：

```
1. 文档信息（版本、作者、状态、修改记录）
2. 背景与目标（为什么做，业务目标，成功指标）
3. 用户与场景（目标用户，核心使用场景）
4. 功能范围（做什么，不做什么）
5. 详细需求（逐功能展开，含流程图、状态机）
6. 非功能需求（性能、安全、兼容性）
7. 验收标准（每个需求点对应的AC）
8. 上线计划与灰度策略
9. 风险与降级方案
10. 附录（原型链接、API文档、数据字典）
```

**AI产品特有章节**（传统PRD中没有）：
- 模型行为规格（期望输出、禁止输出）
- Prompt设计说明（System Prompt策略）
- AI失效降级方案（当模型不可用时的兜底逻辑）
- 评估指标定义（如何判断AI回答的好坏）

### 知识点三：如何写"背景与目标"章节

这是PRD中最容易被写烂的章节，通常被写成："用户反映xxx，所以我们要做xxx功能。" 这种写法缺乏说服力。

正确的写法应遵循 **Why-What-How** 三层结构：

**Why（业务动因）**：
- 市场/竞品触发：竞品A已上线xxx，我们面临xxx风险
- 数据触发：用户留存在Day7跌至xx%，核心原因是xxx
- 战略触发：公司Q3 OKR中有xxx目标需要支撑

**What（产品目标）**：
- 本次要达成的可测量结果（而非功能列表）
- 例：将AI问答的用户7日留存从18%提升至25%

**How（解决思路）**：
- 选择此方案的逻辑（为什么不选其他方案）
- 与其他产品功能的关联和依赖

### 知识点四：验收标准（AC）的写法

验收标准（Acceptance Criteria）是PRD最容易被忽略但最影响交付质量的部分。写法推荐用 **Given-When-Then** 格式：

```
Given（前置条件）：用户已登录，处于AI问答对话界面
When（触发动作）：用户发送一条问题，系统调用AI模型
Then（预期结果）：
  - 正常情况：3秒内返回流式回答，字数不少于50字
  - 超时情况：5秒后显示"正在思考中..."占位符
  - 失败情况：显示降级回答，不暴露错误堆栈
```

AI产品的AC还需要额外定义：
- 模型输出的"好"的标准（如相关性、安全性）
- 敏感词/违禁内容的处理方式
- 最大输出Token数限制

### 知识点五：AI产品PRD中的"记忆管理"写法示例

"记忆管理"功能是AI问答产品的典型复杂功能，PRD写法示例：

**功能描述**：用户可以查看、编辑和删除AI对当前用户的"记忆条目"，AI在后续对话中会参考这些记忆生成个性化回答。

**核心流程**：
```
用户对话 → AI自动提取记忆 → 写入记忆库 → 用户查看"我的记忆"页面 
→ 可编辑/删除 → 下次对话时AI读取记忆 → 注入SystemPrompt → 生成个性化回答
```

**边界说明（Non-Goals）**：
- 本期不做记忆分类/标签功能
- 不支持用户主动添加记忆（只能编辑AI提取的记忆）
- 记忆条目上限为50条（超出时自动删除最旧条目）

### 知识点六：PRD版本管理与变更控制

PRD是活文档，但必须有严格的变更管理，否则会引发"需求蔓延"和"范围膨胀"：

| 变更类型 | 处理方式 |
|---------|---------|
| 澄清性修改（补充说明） | 直接更新，注明修改时间 |
| 范围扩展（新增功能） | 需走需求评审，更新版本号 |
| 范围缩减（砍掉功能） | 需通知所有相关方，记录原因 |
| 需求反转（逻辑改变） | 必须重新走评审流程 |

PRD版本号建议使用语义化版本：`v1.0.0`（主版本.功能版本.修订版本）

## 关键概念详解

### PRD（Product Requirements Document）
**是什么**：描述一个产品功能"应该做什么"的正式文档，是产品经理的核心输出物之一。
**为什么重要**：没有PRD，团队各自理解需求，交付出来的东西往往不是PM想要的；有了PRD，减少80%的返工沟通成本。
**具体例子**：你的AI问答产品要上线"历史会话搜索"功能，PRD需要说清楚：搜索范围（全部历史 vs 近3个月）、搜索方式（关键词 vs 语义搜索）、排序逻辑（时间 vs 相关性）、空状态设计，以及当搜索服务不可用时如何降级。

### Non-Goals（不做清单）
**是什么**：明确列出本次需求范围内不包含的功能或场景。
**为什么重要**：它比"做什么"更难写，也更有价值。明确不做什么，能防止需求在开发过程中被随意扩展，也帮助团队聚焦资源。
**具体例子**：AI问答记忆管理PRD中明确写"本期不支持跨设备记忆同步"，可以防止研发在做方案时默认要做跨设备同步，节省至少2周的开发工作量。

### 验收标准（Acceptance Criteria）
**是什么**：定义一个需求"完成"的可测量、可验证的条件集合。
**为什么重要**：AC是开发和测试的终点线。没有AC，"完成"的定义会因人而异，导致反复返工。
**具体例子**：AI问答回答速度的AC："在P95情况下，首个Token输出时间不超过2秒；完整回答输出时间不超过15秒；超时后显示降级提示，不显示空白。"

### 状态机（State Machine）
**是什么**：描述功能在不同状态之间转换规则的图表，通常用于有多种状态的功能。
**为什么重要**：AI产品中消息状态复杂（发送中→生成中→完成/失败/被中断），不画状态机极易遗漏边界处理。
**具体例子**：AI消息状态机：`空白 → 输入中 → 发送中 → AI生成中 → 完成 | 超时 | 被中断 | 内容违规`，每个状态对应不同的UI展示和用户可操作行为。

## 练习任务输出

**题目**：为AI问答的「记忆管理」功能写PRD大纲

**参考框架**：

```markdown
## 记忆管理功能 PRD 大纲 v1.0

### 1. 背景与目标
- 业务背景：用户每次对话需重复提供个人背景信息，降低使用效率
- 产品目标：通过AI自动记忆提升对话个性化程度，7日留存提升xx%
- 成功指标：记忆功能开启率 > 40%；带记忆对话满意度 > 无记忆高10%

### 2. 用户与场景
- 目标用户：高频使用AI问答的深度用户（每周对话 > 5次）
- 核心场景：
  - 场景A：用户希望AI记住自己的职业背景
  - 场景B：用户想删除AI记错的信息
  - 场景C：用户想看AI目前记住了什么

### 3. 功能范围（In Scope）
- [ ] AI自动从对话中提取记忆条目
- [ ] 用户查看记忆列表页面
- [ ] 用户编辑单条记忆
- [ ] 用户删除单条记忆
- [ ] 对话时AI读取记忆注入上下文

### 4. 不做清单（Non-Goals）
- 本期不做：用户主动添加记忆
- 本期不做：记忆分类/标签
- 本期不做：跨设备同步

### 5. 详细需求（待填写）
[按功能点展开，含流程图]

### 6. 验收标准（待填写）
[Given-When-Then格式]

### 7. 风险与降级
- 风险：记忆提取误差导致用户体验下降
- 降级：提供"清除所有记忆"一键操作
```

## 面试话术

**Q：请介绍一下你写PRD的方法和流程？**
> 我的PRD写作遵循"先对齐再细化"原则：先用一页纸写清楚背景、目标和成功指标，拉关键stakeholder对齐，再展开详细需求。对AI产品，我会额外增加"模型行为规格"和"降级方案"章节，因为AI的不确定性需要在文档中提前定义好处理边界。

**Q：你如何处理开发过程中的需求变更？**
> 我把变更分成三类：澄清性说明直接更新文档；范围扩展需要重新评估优先级和排期，通知所有相关方；需求反转则必须重新走评审流程。核心原则是"变更可以，但必须显式记录和知会"，避免需求悄悄变大而研发不知道。

**Q：一份好的PRD和差的PRD有什么区别？**
> 好的PRD让研发不来问你也能做出你想要的功能。差的PRD要么过于简略（缺边界case和AC），要么过于详细（把设计细节也写进去限制了研发的方案空间）。关键在于把"什么"写清楚，把"怎么做"留给研发。
