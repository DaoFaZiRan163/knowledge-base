## 今日学习内容

> 产品路线图（Product Roadmap）是PM对产品未来方向的可视化承诺，既要对外传递产品愿景，也要对内指导团队排期。然而路线图最大的误区是把它做成"功能交付时间表"——这会让路线图变成枷锁，而非指南。现代产品管理更推崇基于目标的路线图（Outcome-based Roadmap），聚焦"要达成什么"而非"要交付什么"。对AI产品而言，因为技术不确定性高，路线图需要更多弹性空间和明确的假设依赖。

---

## 核心知识点

### 知识点一：路线图的三种形态

不同场景下需要不同形态的路线图：

| 类型 | 时间颗粒度 | 受众 | 核心内容 |
|------|-----------|------|---------|
| 战略路线图 | 季度/年度 | 高层管理、投资人 | 方向、目标、里程碑 |
| 功能路线图 | 月/季度 | 产品、研发、设计 | 功能规划、优先级 |
| Sprint路线图 | 周/双周 | 研发团队 | 具体任务、交付物 |

一个PM通常需要维护两个层级的路线图：面向管理层的战略路线图（季度粒度）和面向研发的功能路线图（月度粒度）。

### 知识点二：Now / Next / Later框架

**Now / Next / Later** 是最推荐初学者上手的路线图框架，核心思路是按确定性而非时间来分层：

```
┌─────────────────────────────────────────────────────────┐
│  NOW（当前Sprint/本月）                                    │
│  正在做的事，需求已确定，方案已评审，研发已启动               │
│  示例：流式回答优化、记忆列表UI、Bad Case标注功能             │
├─────────────────────────────────────────────────────────┤
│  NEXT（下1-2个月）                                        │
│  计划中的事，方向已确定，需求待细化                          │
│  示例：记忆自动提取v2、多轮对话优化、移动端适配               │
├─────────────────────────────────────────────────────────┤
│  LATER（3个月以后）                                       │
│  探索中的方向，还需要验证假设                               │
│  示例：多模态输入、AI主动推送、企业版功能                    │
└─────────────────────────────────────────────────────────┘
```

**为什么不用时间轴**：时间轴路线图承诺了不该承诺的事，一旦需求变化或技术受阻，整张图都需要更新，严重降低团队对路线图的信任度。

### 知识点三：基于目标 vs 基于功能的路线图

**❌ 基于功能的路线图（常见但有问题）**：
```
Q1: 记忆管理功能
Q2: 多轮对话优化  
Q3: 搜索功能
Q4: 企业版
```

问题：只说"做什么"，不说"为什么做"，无法评估做这些事情是否在推动产品目标。

**✅ 基于目标的路线图（现代最佳实践）**：
```
Q1 目标：提升7日留存从18% → 25%
  └── 记忆管理功能（假设：个性化体验提升留存）
  └── 会话搜索（假设：快速找到历史内容减少重复提问）
  
Q2 目标：提升新用户激活率（首次7天完成5次对话）
  └── 引导式首次对话流程
  └── 话题推荐模块
```

基于目标的路线图让每个功能都有明确的"存在理由"，当资源不足时更容易做取舍决策。

### 知识点四：路线图的三个核心决策

做路线图时需要回答三个关键问题：

**1. 这件事解决了哪个用户问题？**
不要直接从"功能想法"出发，先问清楚用户痛点是什么。

**2. 我们如何验证这个假设？**
路线图中的每个功能都基于某个假设（"做了X，用户会做Y，结果是Z"），这些假设需要被明确写出来。

**3. 成功的样子是什么？**
每个路线图条目需要对应一个成功指标，否则做完之后无法判断是否成功。

**AI产品路线图的额外考量**：
- 技术成熟度（模型能力是否支持）
- API成本（功能上线后的成本模型）
- 数据飞轮（这个功能是否帮助积累训练数据）
- 合规要求（AI监管是否影响功能形态）

### 知识点五：优先级排序方法对比

| 方法 | 适用场景 | 核心逻辑 |
|------|---------|---------|
| RICE评分 | 功能较多，需要量化排序 | (触达×影响×信心)÷努力 |
| ICE评分 | 快速决策，轻量版RICE | 影响×信心÷努力 |
| MoSCoW | 需求取舍，快速分类 | 必须/应该/可以/不做 |
| Kano模型 | 理解用户满意度结构 | 基本型/期望型/兴奋型 |
| 机会评分 | 发现未被满足的需求 | 重要性÷满意度 |

路线图优先级排序时，RICE/ICE适合横向比较多个候选功能，MoSCoW适合快速做取舍决策。

### 知识点六：路线图的沟通与维护

路线图不是一次性产出，需要持续维护和主动沟通：

**更新频率建议**：
- Now层：每个Sprint更新一次
- Next层：每月审视一次
- Later层：每季度审视一次

**关键沟通原则**：
- 对外（管理层/投资人）：强调目标和方向，不承诺具体功能的交付时间
- 对内（研发/设计）：强调优先级和依赖关系，提前识别技术风险
- 对业务（销售/客户成功）：说明哪些客户需求在路线图中，哪些不在

**路线图中的"NO"怎么说**：
> "这个需求很有价值，我们也在考虑。但目前我们的优先级是[目标A]，因为[数据依据]。这个需求放在Later层，当我们完成[目标A]后会重新评估。"

## 关键概念详解

### 产品路线图（Product Roadmap）
**是什么**：产品未来一段时间内的方向规划，以可视化方式呈现产品目标和功能演进路径。
**为什么重要**：路线图是PM对齐各方预期的工具——团队知道要做什么、管理层知道资源投向哪里、用户知道产品会变成什么样。没有路线图，团队容易被短期需求淹没，失去方向感。
**具体例子**：AI问答产品Q1路线图目标是"提升7日留存"，对应功能是记忆管理和个性化推荐；Q2路线图目标是"开拓企业客户"，对应功能是团队版和API开放。每个季度的功能选择都服务于当期目标，而不是随机选取。

### Now / Next / Later
**是什么**：按确定性而非时间分层的路线图框架，Now是正在做的、Next是计划中的、Later是探索中的。
**为什么重要**：相比时间轴路线图，它更诚实地反映了产品规划的不确定性，避免对远期功能做出无法兑现的承诺，同时保留了方向感。
**具体例子**：AI问答的Now层包含"流式回答优化"（已在研发中）、Next层包含"多模态输入"（方向确定，技术方案在评估中）、Later层包含"AI主动推送"（想法阶段，需要验证用户是否需要）。

### 基于目标的路线图（Outcome-based Roadmap）
**是什么**：以要达成的业务成果（而非要交付的功能）为核心组织的路线图。
**为什么重要**：当市场变化或用户反馈与预期不同时，基于目标的路线图允许灵活调整实现路径，而不必推翻整张路线图。这对AI产品尤其重要，因为AI能力的边界经常超出预期或低于预期。
**具体例子**：目标是"提升新用户激活率"，原定方案是"推荐引导话题"。如果A/B测试发现引导话题反而降低激活，可以快速切换为"简化首次对话UI"，而目标不变。

### 技术债务（Technical Debt）在路线图中的处理
**是什么**：为了快速交付而积累的不够完美的技术实现，未来需要花时间重构。
**为什么重要**：技术债务不处理会越来越影响开发速度，PM需要在路线图中为技术债务预留空间，而不是一味追加新功能。
**具体例子**：AI问答的Prompt管理代码已经成为瓶颈，每次修改都需要3天。在路线图的Next层加入"Prompt管理系统重构"（技术债务项），虽然用户看不到变化，但研发效率提升后能更快支撑后续功能开发。

## 面试话术

**Q：你是如何规划和维护产品路线图的？**
> 我使用Now/Next/Later框架，以业务目标而非功能列表为核心。每个路线图条目都要回答三个问题：解决什么用户问题、背后的假设是什么、成功指标是什么。对AI产品，我还会额外标注技术成熟度和API成本影响。路线图每月与团队对齐一次，每季度向管理层汇报。

**Q：当资源有限时，你如何决定哪些功能优先做？**
> 我用RICE或ICE模型量化评分，但评分只是起点，最终决策还要考虑战略对齐（是否支撑当期OKR）、技术依赖（哪个功能解锁了后续功能）、以及机会窗口（市场时机）。对于AI产品，我还会特别权衡技术风险：高不确定性的AI功能要么提前做技术验证，要么排在路线图后期，避免阻塞整体交付节奏。
